Hands-on Projects

Project · Resume Screening RAG App

Match resumes to a job description with retrieval + structured extraction, returning evidence-backed scores — and explicitly handling bias/fairness, which interviewers will probe.

Has a fairness angle

What this project proves in an interview

You can combine extraction + retrieval for a real business task AND reason about fairness, explainability and the limits of automated decisions.

Architecture

JD + resumes
Extract structured fields
Retrieve/Match evidence
Score with citations
Human review

Tech stack options

ConcernTool
ExtractionSchema-based (skills, experience)
MatchingEmbedding similarity + criteria checks
ExplainabilityEvidence snippets per score

API design

POST /screen → {jd, resume} ⇒ {match_score, evidence:[...], gaps:[...]}. Never an auto-reject — assistive only.

Data flow

JD → required criteria. Resume → structured extraction + embedding. Match each criterion to resume evidence → score with citations → present to a human reviewer.

Evaluation plan

Agreement with human reviewers on a labeled set; plus a fairness check (does score correlate with protected-proxy attributes it shouldn't?).

Failure modes

  • Bias amplification → audit, blind fields, human-in-the-loop
  • Keyword gaming → semantic + evidence checks
  • Over-trust → assistive only, never auto-reject

Demo script

Screen a resume against a JD → show per-criterion evidence and gaps. Show that it ranks/assists but a human decides.

GitHub README structure

Problem & users · Architecture · Quickstart · Design decisions · Evaluation · Limitations & next steps.

Interview explanation template

“A resume–JD matcher that returns evidence-backed scores per criterion. I deliberately scoped it as assistive with human review, and I audit it for bias — automated hiring decisions are a liability.”

Questions the interviewer may ask about this project (4)

QHow would you explain the Resume Screening RAG project in an interview?IntermediatebehavioralSeniorfreq: highBoth
Model answer

Lead with the problem and the metric: “It matches resumes to a JD with evidence-backed, explainable scores to assist (not replace) a human reviewer.” Then the architecture in one breath, then the one hard tradeoff you made and how you evaluated it.

Strong senior-level answer

I structure it as: problem → users → architecture → the hardest decision and why → how I measured success → what I'd do next. I keep it to 2 minutes and let the interviewer pull threads.

Likely follow-ups
  • What was the hardest part?
  • What would you change with more time?
  • How did you know it worked?
Red-flag answer pattern
Listing technologies with no problem, metric, or tradeoff.
QWhat are the fairness/ethical risks here and how do you handle them?SeniorbehavioralSenior/Stafffreq: emergingBoth
Model answer

Risk of amplifying bias in historical data, proxy discrimination, and over-trust in scores. Mitigate with human-in-the-loop (assistive only), explainable evidence per score, auditing scores against protected proxies, and never auto-rejecting.

Strong senior-level answer

I'd push back on full automation entirely — the system assists ranking with transparent evidence and gets audited for disparate impact. Naming the legal/ethical exposure and designing around it is the point; a candidate who automates rejection silently is the red flag.

Likely follow-ups
  • How would you audit for bias?
  • Would you ever auto-reject?
Red-flag answer pattern
Building an auto-reject pipeline with no fairness consideration.
QHow do you make the match scores explainable and trustworthy?Intermediatetechnical deep-diveSeniorfreq: mediumBoth
Model answer
  • Score per JD criterion, not one opaque number.
  • Attach the resume evidence (quoted snippet) behind each criterion score.
  • Show gaps explicitly so a human can judge.
Strong senior-level answer

A senior makes every score traceable to evidence and frames the tool as assistive ranking — a human always decides.

Likely follow-ups
  • Why per-criterion instead of one score?
Red-flag answer pattern
A single black-box 'match %'.
QHow do you evaluate this without a clean ground truth?Advancedtechnical deep-diveSeniorfreq: emergingBoth
Model answer
  • Measure agreement with human reviewers on a labeled set.
  • Run a fairness check: does the score correlate with attributes it shouldn't (proxies for protected classes)?
  • Track reviewer override rate.
Strong senior-level answer

A senior treats fairness auditing as a first-class metric alongside accuracy, and would refuse to ship full automation.

Likely follow-ups
  • What's a proxy attribute?
Red-flag answer pattern
Only measuring 'accuracy' against past hiring decisions (which encode bias).

Build & ship checklist

  • Structured extraction + evidence
  • Assistive only (human decides)
  • Bias audit
  • Agreement-with-human eval

Project · SQL Agent

A natural-language → SQL agent over a real schema. The canonical 'agent with a dangerous tool' project — so it's really about safety and correctness.

Agent + safety

What this project proves in an interview

You can build an action-taking agent safely: read-only, schema-aware, validated, with guardrails on a tool that could do damage.

Architecture

NL question
Agent: inspect schema
Generate SQL
Validate (read-only, limits)
Execute + summarize

Tech stack options

ConcernTool
DBPostgres / SQLite
SafetyRead-only role, query allow-list, row limits
AgentTool-calling loop / LangGraph

API design

POST /query → {question} ⇒ {sql, rows, summary}. Return the generated SQL for transparency.

Data flow

Question → provide schema to model → generate SQL → validate (SELECT-only, add LIMIT, block DDL/DML) → execute on read-only connection → summarize results.

Evaluation plan

Golden set of question→expected SQL/result. Measure execution accuracy and safety (zero unsafe queries pass the guard).

Failure modes

  • Generating destructive SQL → read-only + validation
  • Hallucinated columns → ground in schema, validate
  • Huge result sets → enforce LIMIT
  • Injection via question → validate, parameterize

Demo script

Ask an analytical question → see the SQL + results. Try to make it DROP a table → guard blocks it.

Interview explanation template

“A text-to-SQL agent on a read-only connection with query validation. The real work is safety: it can never run anything but a bounded SELECT, and I show the SQL for trust.”

Questions the interviewer may ask about this project (4)

QHow would you explain the SQL Agent project in an interview?IntermediatebehavioralSeniorfreq: highBoth
Model answer

Lead with the problem and the metric: “It turns natural-language questions into safe, read-only SQL over a real schema and summarizes the results.” Then the architecture in one breath, then the one hard tradeoff you made and how you evaluated it.

Strong senior-level answer

I structure it as: problem → users → architecture → the hardest decision and why → how I measured success → what I'd do next. I keep it to 2 minutes and let the interviewer pull threads.

Likely follow-ups
  • What was the hardest part?
  • What would you change with more time?
  • How did you know it worked?
Red-flag answer pattern
Listing technologies with no problem, metric, or tradeoff.
QHow do you make a SQL-generating agent safe?Advancedsystem designSeniorfreq: highBoth
Model answer

Read-only DB role, validate generated SQL (SELECT-only, no DDL/DML), enforce LIMITs and timeouts, parameterize, and surface the SQL for transparency. Treat the question as untrusted.

Strong senior-level answer

Safety lives in what the tool can't do, not in the prompt: a least-privilege role makes destructive queries impossible regardless of what the model emits. I also add row/cost limits and log every query for audit.

Likely follow-ups
  • What if it needs write access?
  • How do you ground it in the schema?
Red-flag answer pattern
Giving the agent a read-write connection and trusting the prompt.
QHow do you get correct SQL on a large or ambiguous schema?Advancedtechnical deep-diveSeniorfreq: highBoth
Model answer
  • Give the model the relevant schema (table/column descriptions) — retrieve it (RAG over the schema) rather than dumping everything.
  • Provide example question→SQL pairs (few-shot).
  • Self-check / repair: on an execution error, feed it back and retry once.
Strong senior-level answer

A senior invests in a semantic layer of approved metrics/joins — it removes most ambiguity and is the highest-leverage fix — and measures execution accuracy on a golden set.

Likely follow-ups
  • What's a semantic layer?
  • How do you handle ambiguous metrics?
Red-flag answer pattern
Dumping the whole schema and hoping.
QHow would you explain this project's value in an interview?IntermediatebehavioralSeniorfreq: highBoth
Model answer

Lead with the safety story: 'A text-to-SQL agent on a read-only connection with query validation — the interesting engineering is that it physically cannot run anything destructive, and it shows the SQL for trust.'

Strong senior-level answer

Likely follow-ups

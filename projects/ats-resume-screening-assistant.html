Project summary

Built a GenAI-powered ATS assistant that compares resumes against job descriptions, extracts candidate skills and experience, ranks candidates using semantic matching, explains the match score, and helps HR teams shortlist candidates with human review.

Evaluation plan

  • Create 20–50 sample JD–resume pairs; manually label strong / medium / weak.
  • Compare the system's ranking with the human ranking.
  • Track extraction accuracy, hallucinated skills, and missing-skill correctness.
  • Track explanation quality, false positives and false negatives.
  • Add a human feedback loop so labels grow over time.

Responsible AI & guardrails

⛔
HR is high-risk — design for it
  • Do not make final hiring decisions automatically — keep the recruiter in the loop.
  • Do not use or rank on protected attributes: gender, age, religion, caste, race, photo, address, marital status.
  • Mask or ignore PII where possible.
  • Explain every score; store an audit trail of screening decisions.
  • Allow recruiter override; clearly mark the tool as decision-support, not decision-maker.

Failure modes

  • Resume parsing errors; long resumes causing context issues; duplicate resumes.
  • Missing or hallucinated skills; overweighting keyword matches; underweighting transferable skills.
  • Bias from historical hiring data; poor JD quality.
  • Candidate ranking instability; LLM output-format failures.

How to explain it in interviews

“I built an ATS Resume Screening Assistant as a practical GenAI project for HR candidate shortlisting. The system takes a job description and multiple resumes, extracts structured information like skills, experience, education, projects, and tools, then compares candidates against the job description using embeddings and LLM-based reasoning.

Q2How do you match resumes against job descriptions?Advancedtechnical deep-diveSenior
Why the interviewer asks this

Probes whether your matching is principled (semantic + structured) or naive keyword overlap.

Model answer
  • Extract structured requirements from the JD (skills, must-haves, experience).
  • Extract a structured profile from each resume.
  • Embed and semantically match resume content to JD requirements (hybrid: embeddings + keyword for exact tools/certs).
  • Score per requirement, then aggregate into an overall match with the LLM explaining the gaps.
Strong senior-level answer

I score per-requirement with evidence rather than one opaque number, combine semantic similarity with exact-term matching (a specific cert/tool must match exactly), and keep the LLM's role to explanation/aggregation over retrieved evidence — not free-form judgment.

Likely follow-ups
  • Semantic vs exact for a required certification?
  • How do you weight must-have vs nice-to-have?
Red-flag answer pattern
Pure keyword overlap, or letting the LLM 'decide' with no structured evidence.
What to mention from the project
  • Per-requirement scoring with evidence
  • Hybrid semantic + exact matching
How to connect this to GenAI roles
🔗

Hybrid retrieval + reranking + grounded LLM reasoning — directly transferable to any RAG product.

Q3Why use embeddings instead of only keyword matching?Intermediatetechnical deep-diveMid–Senior
Why the interviewer asks this

Checks you understand semantic matching and its limits.

Model answer

Keyword matching misses paraphrase and transferable skills — 'built REST APIs' vs 'developed backend services', or 'PyTorch' experience for a 'deep learning' requirement. Embeddings capture meaning, so a strong candidate phrased differently isn't dropped.

But you still need keyword/exact matching for hard requirements like a specific certification or tool — so the answer is hybrid.

Strong senior-level answer

I'd note embeddings can over-match loosely related text, so I keep exact matching for must-haves and use embeddings for transferable/semantic fit — and validate the mix on labeled JD-resume pairs.

Likely follow-ups
  • When does semantic matching hurt you?
  • How do you handle synonyms for tools?
Red-flag answer pattern
Claiming embeddings alone solve everything, ignoring exact-match needs.
What to mention from the project
  • Transferable-skill matching
  • Why hybrid (exact + semantic)
How to connect this to GenAI roles
🔗

This is the dense-vs-sparse-vs-hybrid retrieval discussion every RAG interview has.

Q4How do you prevent hallucinated skills?Advancedtechnical deep-diveSenior
Why the interviewer asks this

HR + hallucination = real harm; tests grounding discipline.

Model answer
  • Ground extraction strictly in the resume text — extract only what's present, with the source span.
  • Use structured output (schema) so the model returns fields, not prose.
  • Verify each extracted skill maps to evidence in the resume; drop unsupported ones.
  • Require citations/evidence in the match explanation.
Strong senior-level answer

I add a verification pass that checks each claimed skill against the source text and flags anything unsupported, and I treat a hallucinated skill as a tracked error rate in evaluation — not an occasional annoyance.

Likely follow-ups
  • How do you measure the hallucination rate?
  • What if the resume is ambiguous?
Red-flag answer pattern
Trusting the LLM's extracted skills with no grounding or verification.
What to mention from the project
  • Evidence-grounded extraction
  • Verification pass
  • Hallucinated-skill rate as a metric
How to connect this to GenAI roles
🔗

Grounding + faithfulness verification is the core RAG anti-hallucination pattern.

Q5How do you evaluate the ranking quality?Advancedtechnical deep-diveSenior
Why the interviewer asks this

Separates demo-builders from engineers who measure quality.

Model answer
  • Build 20–50 JD-resume pairs and have humans label strong/medium/weak.
  • Compare the system's ranking to the human ranking (rank correlation / agreement).
  • Track extraction accuracy, hallucinated-skill rate, missing-skill correctness, and explanation quality.
  • Track false positives (weak candidate ranked high) and false negatives.
Strong senior-level answer

I optimize for not missing strong candidates (false negatives are costly) and treat the labeled set as a regression gate. I'd also run a fairness check that scores don't correlate with protected proxies.

Likely follow-ups
  • Which error is worse here, FP or FN?
  • How do you keep the labeled set fresh?
Red-flag answer pattern
'It looked reasonable on a few resumes.'
What to mention from the project
  • Human-labeled JD-resume pairs
  • Ranking agreement + error rates
  • Fairness check
How to connect this to GenAI roles
🔗

Golden datasets + LLM/eval metrics + human calibration — the LLM-evaluation skill set.

Q6What guardrails are needed for an HR use case?Seniorsystem designSenior/Staff
Why the interviewer asks this

HR is high-risk/regulated; tests responsible-AI maturity.

Model answer
  • Decision-support only — never auto-reject/hire; recruiter makes the call.
  • Ignore/mask protected attributes (gender, age, religion, caste, race, photo, address, marital status).
  • Mask PII where possible.
  • Explain every score; store an audit trail; allow recruiter override.
Strong senior-level answer

I'd design the system so protected attributes never enter the features, run disparate-impact audits, keep humans in the loop on every decision, and log decisions for auditability — and I'd refuse to ship full automation.

Likely follow-ups
  • How do you audit for bias?
  • Would you ever auto-reject?
Red-flag answer pattern
Auto-rejecting candidates, or using/ranking on protected or proxy attributes.
What to mention from the project
  • Decision-support not decision-maker
  • Protected-attribute exclusion + audit trail
  • Recruiter override

Q7How would you design the FastAPI backend?Advancedsystem designSenior
Why the interviewer asks this

Tests production API design around a GenAI pipeline.

Model answer

Endpoints: create job (POST /api/jobs), upload resumes (POST /api/resumes/upload), run screening (POST /api/screen), list ranked candidates (GET /api/jobs/{id}/candidates), candidate scorecard (GET /api/candidates/{id}/scorecard), recruiter feedback (POST /api/feedback).

Separate layers: parsing/extraction, embedding/retrieval, scoring/explanation, and a thin API layer with structured (validated) JSON responses, async for long screening runs, and persistence in Postgres/SQLite.

Strong senior-level answer

I'd make screening async (job queued, results polled/streamed), validate all LLM output against schemas with a repair loop, store an audit trail, and keep the LLM/vector providers behind a swappable interface.

Likely follow-ups
  • How do you handle a long screening run?
  • How do you make LLM output reliable?
Red-flag answer pattern
One synchronous endpoint doing parse+embed+LLM with no validation or persistence.
What to mention from the project
  • The 6 endpoints + their purpose
  • Async screening
  • Schema-validated JSON
How to connect this to GenAI roles
🔗

Production GenAI API design — exactly what applied GenAI roles build.

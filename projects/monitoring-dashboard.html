Project · Production GenAI Monitoring Dashboard

The LLMOps capstone: a dashboard over traces of a live RAG/agent app showing cost, latency, quality scores and drift, with alerts. Proves the operational maturity platform teams want.

Platform signal

What this project proves in an interview

You operate GenAI like a real service: end-to-end tracing, cost/latency/quality monitoring, drift detection and alerting.

Architecture

App requests
Trace spans → store
Score sample (judge)
Dashboard + alerts

Tech stack options

ConcernTool
TracingLangfuse / OpenTelemetry / DIY spans
ScoringLLM-as-judge on sampled traces
DashboardGrafana / Streamlit / custom
AlertsThreshold + anomaly

API design

Instrument the existing RAG/agent app to emit spans (retrieval, LLM, tools, tokens, cost, latency); dashboard reads the trace store.

Data flow

Each request emits a trace tree → stored → a sampler runs faithfulness/quality judges → metrics aggregated (cost/req, p95 latency, abstain rate, quality) → dashboard + alerts on regressions/drift.

Evaluation plan

The dashboard IS partly the eval surface; validate that injected regressions (e.g. degraded retrieval) actually trip the quality alert.

Failure modes

  • PII in traces → redaction
  • Sampling misses rare failures → adjust sampling
  • Alert fatigue → tune thresholds, anomaly detection

Demo script

Show a live trace drill-down (why was this answer slow/expensive/bad), the cost/latency/quality charts, and trigger an alert by degrading retrieval.

Interview explanation template

“A monitoring dashboard over my RAG app's traces: cost/req, p95 latency, faithfulness on sampled traffic, and drift alerts. I can drill into any request to see retrieval, prompts and tokens — which is how I debug quality in production.”

Questions the interviewer may ask about this project (4)

QHow would you explain the Monitoring Dashboard project in an interview?IntermediatebehavioralSeniorfreq: highBoth
Model answer

Lead with the problem and the metric: “It traces every request and surfaces cost, latency, quality and drift with alerts, so production issues are visible and debuggable.” Then the architecture in one breath, then the one hard tradeoff you made and how you evaluated it.

Strong senior-level answer

I structure it as: problem → users → architecture → the hardest decision and why → how I measured success → what I'd do next. I keep it to 2 minutes and let the interviewer pull threads.

Likely follow-ups
  • What was the hardest part?
  • What would you change with more time?
  • How did you know it worked?
Red-flag answer pattern
Listing technologies with no problem, metric, or tradeoff.
QWhat would you monitor for a production GenAI app and why?Advancedsystem designSeniorfreq: highBoth
Model answer

Cost per request, p95 latency, error/abstain rates, sampled quality (faithfulness/relevance), and input/output drift — plus per-step traces to debug. Alert on regressions and anomalies.

Strong senior-level answer

I tie monitoring to the eval set: the same judges that gate deploys score sampled production traffic, so quality is continuously measured, not just at release. I attach trace ids end-to-end and log retrieved context so most debugging is a single trace lookup. Drift detection on inputs warns me before quality drops.

Likely follow-ups
  • How do you detect quality drift without labels?
  • How do you handle PII in traces?
Red-flag answer pattern
Monitoring only uptime/latency and ignoring quality.
QWalk me through using this dashboard to debug a bad answer.Advancedtechnical deep-diveSeniorfreq: highBoth
Model answer

Open the request's trace and check in order: retrieved chunks (right ones?), the actual prompt, the model output, then any guardrail step. The logged context tells you instantly whether it was a retrieval or generation failure.

Strong senior-level answer

A senior designed the trace specifically so this takes seconds, then turns the bad case into a golden-set regression test.

Likely follow-ups
  • What do you log to make this fast?
Red-flag answer pattern
Having no per-step trace to inspect.
QHow do you detect quality regressions in production without labels?Advancedsystem designSeniorfreq: mediumBoth
Model answer
  • Run an LLM-as-judge faithfulness check on a sample of live traffic; alert on score drops.
  • Watch proxy signals: abstain rate, citation coverage, thumbs-down, latency.
  • Detect input drift (new kinds of questions).
Strong senior-level answer

A senior reuses the same judges that gate deploys to score production, so offline and online quality share one definition.

Likely follow-ups
  • How do you trust the judge at scale?
Red-flag answer pattern
Assuming no labels means quality can't be measured.

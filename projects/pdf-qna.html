Project · PDF Q&A App

RAG specialized to PDFs: parsing, layout-aware chunking, page-level citations. The most common 'show me a RAG demo' ask, done with rigor.

What this project proves in an interview

You can handle messy real documents (tables, headers, multi-column) — not just clean text — and cite back to the exact page.

API design

POST /upload (pdf) → indexed; POST /ask → {answer, citations:[{page, snippet}]}.

Data flow

PDF → parse text + page numbers → chunk preserving structure → embed → index with page metadata → retrieve → answer citing pages.

Evaluation plan

Golden set of Q→(answer, page). Measure answer correctness + citation-page accuracy.

Failure modes

  • Tables/multi-column mangled by naive parsing
  • Citations point to wrong page
  • Scanned PDFs need OCR

Demo script

Upload a real PDF, ask a question whose answer is in a table, get the right answer with the right page citation.

Questions the interviewer may ask about this project (4)

QHow would you explain the PDF Q&A project in an interview?IntermediatebehavioralSeniorfreq: highBoth
Model answer

Lead with the problem and the metric: “Upload a PDF, ask questions, get answers cited to the exact page.” Then the architecture in one breath, then the one hard tradeoff you made and how you evaluated it.

Strong senior-level answer

I structure it as: problem → users → architecture → the hardest decision and why → how I measured success → what I'd do next. I keep it to 2 minutes and let the interviewer pull threads.

Likely follow-ups
  • What was the hardest part?
  • What would you change with more time?
  • How did you know it worked?
Red-flag answer pattern
Listing technologies with no problem, metric, or tradeoff.
QWhat's hard about RAG over PDFs specifically?Intermediatetechnical deep-diveSeniorfreq: mediumBoth
Model answer

Parsing: PDFs have tables, multi-column layouts, headers/footers and sometimes are scanned. Naive text extraction destroys structure and pollutes chunks; you need layout-aware parsing (and OCR for scans) plus page metadata for citations.

Strong senior-level answer

I evaluate parsing quality separately because most PDF-RAG failures are upstream of retrieval. I keep page/section metadata through the pipeline so citations are precise and access control is possible.

Likely follow-ups
  • How do you handle scanned PDFs?
  • How do you cite a table cell?
Red-flag answer pattern
Assuming PDF = clean text.
QHow do you evaluate this PDF Q&A app?Intermediatetechnical deep-diveSeniorfreq: highBoth
Model answer

Build a golden set of question → (answer, page). Measure answer correctness AND citation-page accuracy separately. Add 'not in the document' cases to test the abstain path.

Strong senior-level answer

A senior evaluates parsing quality upstream too (did the table survive extraction?), since most PDF-RAG failures are pre-retrieval.

Likely follow-ups
  • How do you test parsing separately?
Red-flag answer pattern
Only checking a few answers look right.
QA user uploads a 300-page PDF — what changes?Advancedsystem designSeniorfreq: mediumBoth
Model answer
  • Chunk and index it; never stuff the whole PDF into the prompt.
  • Show ingestion progress; index asynchronously.
  • Retrieval + rerank keeps only the few relevant chunks per question, so size doesn't blow up cost.
Strong senior-level answer

A senior notes per-document scaling (one big doc) differs from corpus scaling (many docs) and handles both — async ingestion and tight retrieval keep latency/cost flat regardless of document size.

Likely follow-ups
  • How do you keep latency flat as docs grow?
Red-flag answer pattern
Trying to fit the whole PDF in the context window.

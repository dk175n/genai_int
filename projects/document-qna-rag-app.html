Project summary

Built a Document QnA RAG application that allows users to upload or index large collections of PDFs, ask natural-language questions, retrieve the most relevant document chunks, and generate grounded answers with citations. The system is designed for scalable ingestion, efficient retrieval, low-latency responses, hallucination reduction, and production-style monitoring.

How this system handles thousands of PDFs

  • Do not process all PDFs at query time — ingest and index offline/asynchronously.
  • Store chunks and metadata in a vector database; use batch embedding (not one-by-one).
  • Use document-level metadata filters to reduce the search space; use hybrid search when keyword precision matters.
  • Use approximate nearest-neighbor (ANN) search for fast retrieval; rerank only a small candidate set.
  • Cache frequent queries and retrieved contexts; stream the final answer so the user sees output quickly.
  • Use background workers for large ingestion jobs; track ingestion status per document.
  • Store citations and source chunk IDs; monitor retrieval latency, LLM latency, and total response time.

How to reduce latency and lag

Latency is reduced by moving expensive work out of the user query path. Document parsing, chunking, embedding generation, and indexing should happen asynchronously before the user asks a question. At query time, the system should only embed the query, retrieve a small candidate set from the vector DB, optionally rerank top results, build a compact prompt, and stream the LLM response. Additional optimizations: Redis caching, metadata filtering, ANN indexes, smaller/faster embedding models, prompt compression, top-k tuning, model fallback, and monitoring p95 latency.

Failure modes

  • Bad PDF extraction; scanned PDFs without OCR; poor chunking; missing metadata.
  • Embedding model mismatch; vector DB returns semantically similar but wrong chunks.
  • Reranker latency too high; too many chunks in prompt; LLM hallucination; missing citations.
  • Stale document index; slow ingestion; query timeout; high token cost; user asks outside available documents.

Guardrails & security

⛔
Document Q&A needs real guardrails
  • Answer only from retrieved documents where required; say “I could not find this in the documents” when evidence is missing.
  • Show citations for every factual claim.
  • Restrict access by collection or user role; never expose documents across users; protect sensitive documents.
  • Validate file type/size; scan uploads if needed.
  • Log prompts/responses carefully; don't store sensitive data in traces unless masked.
  • Handle prompt injection hidden inside documents (treat document text as untrusted).

How to explain it in interviews

📝
90-second answer

“I built a Document QnA RAG application for answering questions over large PDF collections. The system ingests PDFs, extracts text, chunks the content with metadata, generates embeddings, stores them in a vector database, and retrieves the most relevant chunks when a user asks a question. The LLM then generates an answer grounded in the retrieved context and returns citations to the original source documents.

The important design point is that expensive work like parsing, chunking, embedding, and indexing is done asynchronously before query time. At query time, the app only embeds the user question, performs vector or hybrid retrieval, optionally reranks the top candidates, builds a compact prompt, and streams the answer. This keeps the system usable even when the document collection is large.

From an interview perspective, this project demonstrates RAG architecture, vector search, chunking strategy, retrieval optimization, latency reduction, citation-based answering, hallucination control, FastAPI backend design, and production-style monitoring.

Once benchmarked, I would mention the actual tested document count, chunk count, retrieval latency, p50/p95 response latency, and citation accuracy to make the project explanation measurable.”

Q1Explain the Document QnA RAG app end to end.Intermediateproject deep-diveSenior
Why the interviewer asks this

Tests whether you can describe a RAG system as an end-to-end pipeline, separating offline ingestion from online query.

Model answer

Two halves. Offline (ingestion): upload/index PDFs → parse (OCR if scanned) → clean + extract metadata → chunk → batch-embed → store in a vector DB with metadata. Online (query): embed the question → hybrid/vector search with metadata filters → rerank the top candidates → assemble a compact context → LLM generates a grounded answer with citations → stream the response and log a trace.

Strong senior-level answer

I stress the design principle: all expensive work (parse/chunk/embed/index) happens asynchronously before query time, so the query path only does embed → retrieve → rerank → generate → stream. That's what keeps it usable at large scale.

  • What's the trickiest part of ingestion?

Describing it as 'load all PDFs and ask the LLM' — processing documents at query time.

Q5How did you choose chunk size and overlap?Advancedtechnical deep-diveSenior
Why the interviewer asks this

Chunking is the highest-leverage RAG quality decision.

Model answer

Tune against a labeled query set, not by habit. Smaller chunks → precise matching but answers can split; larger → more context but diluted embeddings and wasted window. Start ~400–512 tokens with ~10–20% overlap; respect the embedding model's max length.

Often best: parent-document retrieval — embed small chunks, return the larger surrounding passage.

Strong senior-level answer

I check whether answers span chunk boundaries and measure recall@k across a few sizes, and I keep page/section metadata so citations stay precise.

Likely follow-ups
  • What overlap and why?
  • How does chunk size affect citations?
Red-flag answer pattern
A fixed number with no measurement.
What to mention from the project
  • Measured chunk sizing
  • Parent-document retrieval
  • Metadata for citations

Q7Dense search vs hybrid search — which is better for document Q&A?Advancedtechnical deep-diveSenior
Why the interviewer asks this

Tests understanding of retrieval beyond plain embeddings.

Model answer

Dense (embeddings) captures meaning/paraphrase; sparse (BM25) catches exact terms — IDs, codes, names, acronyms common in documents. Hybrid fuses both and is the production default for document Q&A.

Rerank the fused candidates for precision.

Strong senior-level answer

I weight by content: documents with many exact identifiers (legal/finance/technical) lean more on keyword; conversational FAQ leans dense. I validate the mix on labeled pairs.

Likely follow-ups
  • How do you fuse the two lists?
  • When does dense-only fail?
Red-flag answer pattern
Insisting embeddings alone are always enough.
What to mention from the project
  • Hybrid default for docs

Q8Why and when would you use reranking?Advancedtechnical deep-diveSenior
Why the interviewer asks this

Tests precision/latency tradeoff awareness.

Model answer

First-pass retrieval is fast but imprecise. A cross-encoder reranker reads query+chunk together and reorders the top candidates so only the best few enter the prompt — keeping context small and precise. Run it only on the top ~20–50, not the whole corpus.

Tradeoff: it adds latency, so it's applied to a small candidate set.

Strong senior-level answer

I measure the recall@k / faithfulness lift to justify the added latency, and keep the candidate set small so reranking stays within the latency budget.

Likely follow-ups
  • Why not rerank everything?
  • How much latency does it add?
Red-flag answer pattern
Stuffing many un-reranked chunks into the prompt.
What to mention from the project
  • Rerank small candidate set
  • Smaller, precise context

Q12How do you handle document updates, deletion, and re-indexing?Advancedsystem designSenior
Why the interviewer asks this

Freshness/lifecycle is what separates a one-shot demo from a real system.

Model answer
  • Track each document + its chunks with ids so you can update/delete precisely.
  • On update: re-ingest the changed document, re-embed its chunks, replace them; on delete: remove its chunks (tombstone) so they leave search.
  • Invalidate caches for affected queries; record ingestion status per document.
Strong senior-level answer

I do incremental re-indexing (only the changed document, not the whole corpus) and surface 'last updated' in citations so users can judge recency. Stale index is a silent trust-killer, so I monitor ingestion lag.

Likely follow-ups
  • Full vs incremental re-index?
  • How do you avoid serving deleted-doc chunks from cache?
Red-flag answer pattern
Full re-index of everything on any change, or no deletion path.
What to mention from the project
  • Per-document chunk ids
  • Incremental re-index + tombstoning
  • Cache invalidation

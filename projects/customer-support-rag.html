Project · Customer Support RAG Bot

Production-shaped RAG: answer from a help center with citations, abstain + escalate to a human when unsure, with guardrails and feedback capture. The most 'business-relevant' RAG project.

Production-shaped

What this project proves in an interview

You can ship customer-facing RAG responsibly: grounding, abstain/escalate, guardrails, and a feedback loop — the things businesses actually need.

API design

POST /support → {question, user} ⇒ {answer?, citations?, escalate:bool}. Enforce per-user/tenant access at retrieval.

Data flow

Question → hybrid retrieve over KB (tenant-scoped) → if coverage below threshold → escalate; else grounded answer with citations → output guard → capture user feedback → feed failures into eval set.

Evaluation plan

Golden set + 'deflection' metric (correctly auto-answered vs correctly escalated). Track faithfulness and escalation precision.

Failure modes

  • Confident wrong answer → coverage gate + abstain
  • Cross-tenant leak → retrieval access control
  • Stale KB → re-index pipeline
  • Injection in KB articles → untrusted content

Demo script

In-scope question → cited answer; ambiguous/unknown → graceful escalation; show feedback capture and the eval loop.

Questions the interviewer may ask about this project (4)

QHow would you explain the Customer Support RAG Bot project in an interview?IntermediatebehavioralSeniorfreq: highBoth
Model answer

Lead with the problem and the metric: “It answers support questions from the help center with citations, escalating to a human when it isn't confident.” Then the architecture in one breath, then the one hard tradeoff you made and how you evaluated it.

Strong senior-level answer

I structure it as: problem → users → architecture → the hardest decision and why → how I measured success → what I'd do next. I keep it to 2 minutes and let the interviewer pull threads.

Likely follow-ups
  • What was the hardest part?
  • What would you change with more time?
  • How did you know it worked?
Red-flag answer pattern
Listing technologies with no problem, metric, or tradeoff.
QHow do you decide when the bot should answer vs escalate to a human?Advancedsystem designSeniorfreq: highBoth
Model answer

A coverage/confidence gate on retrieval + answer: if the top retrieval scores or a faithfulness check fall below threshold, abstain and escalate. Tune the threshold against the cost of a wrong answer vs an unnecessary escalation.

Strong senior-level answer

I optimize for trust: in support, a confident wrong answer is worse than an escalation, so I bias toward abstaining and measure escalation precision. I also close the loop — every escalation and thumbs-down becomes an eval case and often a KB gap to fill.

Likely follow-ups
  • How do you prevent cross-tenant data leaks?
  • How do you measure deflection without hurting CSAT?
Red-flag answer pattern
Always answering to maximize 'deflection' regardless of correctness.
QHow do you prevent the bot leaking another customer's or internal-only data?Advancedsystem designSeniorfreq: highBoth
Model answer
  • Scope retrieval to the customer's tenant/locale via metadata filters — restricted docs are never candidates.
  • Keep internal-only articles out of the customer-facing index (or tag and filter them).
  • Output guard scans for internal terms/PII before replying.
Strong senior-level answer

A senior enforces tenancy at the retrieval layer (not the prompt) and may use separate indexes for hard isolation.

Likely follow-ups
  • Separate index vs metadata filter?
Red-flag answer pattern
Trusting the prompt to keep internal info hidden.
QHow do you keep the knowledge base fresh and measure quality over time?Intermediatetechnical deep-diveSeniorfreq: mediumBoth
Model answer
  • Incremental re-index on article changes; show 'last updated' in citations.
  • Track deflection, escalation precision, faithfulness on sampled traffic, and CSAT.
  • Feed every thumbs-down and escalation into the eval set and as a KB-gap signal.
Strong senior-level answer

A senior closes the loop: production failures become eval cases and content-gap tickets, so the bot and the KB improve together.

Likely follow-ups
  • How do you find KB gaps?
Red-flag answer pattern
Static KB with no freshness or feedback loop.

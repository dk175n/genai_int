Project · Build Your First RAG App

The single most important portfolio project. A minimal-but-honest RAG service: ingest documents, retrieve with hybrid search + reranking, answer with citations and an abstain path, and evaluate it. Everything on the RAG Mastery page, made real.

API design

POST /ask → {question} ⇒ {answer, citations:[{source, snippet}], used_context:bool}. POST /ingest → {doc} ⇒ {chunks_indexed}. Keep responses structured so you can evaluate and trace them.

Data flow

Ingest: document → chunks (structure-aware + overlap) → embeddings → vector store with metadata (source, page). Query: embed query → hybrid retrieve (dense + BM25) → rerank top-50 to top-5 → grounded prompt with citations + abstain rule → LLM → validate citations exist → return.

Evaluation plan

Build a 30-case golden set (question → relevant source). Measure retrieval (recall@5, MRR) and generation (faithfulness via LLM-judge, abstain-correctness on out-of-scope questions). Run it in CI; fail on regression. Publish the numbers in the README.

Failure modes

  • Pure semantic retrieval misses exact terms → add BM25/hybrid
  • Chunks too big/small → tune with the golden set
  • Model answers from prior knowledge → enforce context-only + abstain
  • No citations → can't audit; require them
  • Latency from reranking everything → only rerank top candidates

Demo script

1) Ask an in-scope question → correct answer with citations. 2) Ask an out-of-scope question → it abstains. 3) Show the eval dashboard/numbers. 4) Toggle hybrid off to show recall drop. A 3-minute demo that includes the abstain case and the eval numbers beats a flashy chat UI.

GitHub README structure

README sections: Problem & who it's for · Architecture diagram · Quickstart (docker compose up) · Design decisions (chunking, hybrid, rerank — with rationale) · Evaluation (golden set + numbers) · Limitations & next steps. The 'Evaluation' and 'Design decisions' sections are what hiring engineers actually read.

Interview explanation template

“I built a RAG service over . The interesting part wasn't the happy path — it was getting retrieval right (hybrid + rerank lifted recall@5 from X to Y) and making it trustworthy (citations + abstain), which I proved with a 30-case eval that runs in CI.”

Questions the interviewer may ask about this project (3)

QHow would you explain the First RAG App project in an interview?IntermediatebehavioralSeniorfreq: highBoth
Model answer

Lead with the problem and the metric: “It answers questions over a document set with cited, grounded answers, and I measure recall@5 and faithfulness.” Then the architecture in one breath, then the one hard tradeoff you made and how you evaluated it.

Strong senior-level answer

I structure it as: problem → users → architecture → the hardest decision and why → how I measured success → what I'd do next. I keep it to 2 minutes and let the interviewer pull threads.

Likely follow-ups
  • What was the hardest part?
  • What would you change with more time?
  • How did you know it worked?
Red-flag answer pattern
Listing technologies with no problem, metric, or tradeoff.
QWhy hybrid search and reranking instead of just embeddings?Advancedtechnical deep-diveSeniorfreq: highBoth
Model answer

Embeddings miss exact terms (IDs, names); BM25 catches them. Reranking reorders candidates with a cross-encoder for precision so the few chunks in context are the right ones. I measured the recall@5 gain.

Strong senior-level answer

I drove it with a labeled set rather than belief — hybrid + rerank gave a measurable recall and faithfulness lift, and it kept context small (cheaper, less 'lost in the middle'). I'd show the before/after numbers.

Likely follow-ups
  • How did you fuse dense and sparse scores?
  • What reranker and why?
Red-flag answer pattern
Adding components with no measured justification.
QHow do you know the answers are actually grounded?Advancedtechnical deep-diveSeniorfreq: highBoth
Model answer

Citations let me trace each claim to a source, and a faithfulness judge scores whether claims are supported by retrieved context, run over the golden set and sampled in production.

Strong senior-level answer

I gate deploys on faithfulness not dropping, and I added 'should abstain' cases so I measure both correct answers and correct refusals. Groundedness is a number I track, not a hope.

Likely follow-ups
  • How do you trust the judge?
Red-flag answer pattern
'It looked right when I tried it.'

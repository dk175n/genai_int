90-Second Project Story

📝
Say this, out loud, until it's natural

“In the Indorama Digitization project, I worked on backend machine learning and API development for industrial equipment and loop monitoring. The goal was to digitize plant operations by exposing model predictions, anomaly scores, health scores, trip scores, instrumentation scores, and calculated KPIs through backend APIs, which were then integrated with TCG Mcube dashboards.

I worked mainly in Python and FastAPI, where I created APIs for different loops and equipment. The backend pipeline fetched plant time-series data from Elasticsearch or databases, transformed it into model-ready format, calculated KPIs or predictions, generated equipment-level health outputs, and returned structured JSON for dashboard consumption.

The project also involved deployment using Docker and Kubernetes, including API packaging, environment configuration, service deployment, health checks, and integration testing. This project helped me understand end-to-end production ML: data ingestion, feature engineering, model inference, API design, dashboard integration, monitoring, debugging, and deployment.”

Q1Walk me through the Indorama Digitization project end to end.Intermediateproject deep-diveSenior
Why the interviewer asks this

Openers test whether you can compress a real project into a clear, structured narrative and whether you actually owned the work.

Model answer

Lead with the business problem, then the pipeline:

  • Problem — digitize plant operations so operators could monitor equipment health, anomalies and KPIs in real time.
  • Pipeline — fetch time-series data from Elasticsearch/DB → clean/pivot/validate → feature engineering & KPI calculation → ML inference / anomaly logic → produce health/trip/instrumentation scores → expose via FastAPI REST APIs → integrate with TCG Mcube dashboards.
  • Ops — packaged and deployed with Docker/Kubernetes; health checks, env config, debugging.
Strong senior-level answer

I frame it as end-to-end production ML and name the one hard part (e.g. reliable time-series ingestion/validation across many loops) and how I handled scale (hundreds of loop/equipment APIs). I keep it to ~90 seconds and let them pull threads.

Likely follow-ups
  • What was the hardest part?
  • What did you personally own vs the team?
  • How many APIs/equipment types?
Red-flag answer pattern
Listing technologies with no problem statement, no flow, and no clear personal contribution.
What to mention from the project
  • Multiple plant loops & equipment
  • Elasticsearch/DB time-series source
  • Health/trip/instrumentation/anomaly scores + KPIs
  • FastAPI + Mcube dashboard integration
How to connect this to GenAI roles
🔗

Same end-to-end shape as a RAG service: ingest → transform → model → serve via API → integrate with a UI → monitor. You can map each stage directly when asked to design a GenAI system.

Q2How did you design the backend ML API architecture?Advancedtechnical deep-diveSenior
Why the interviewer asks this

Tests whether you can design maintainable backend services around models, not just call a model in a script.

Model answer
  • A data-access layer (Elasticsearch/DB fetch) separated from transform and inference layers.
  • Reusable transform/feature code shared across loops to avoid duplicating logic per endpoint.
  • Per-loop/equipment endpoints returning structured JSON for the dashboard.
  • Config-driven so new loops/equipment could be added without rewriting the stack.
Strong senior-level answer

I'd stress separation of concerns and reuse: hundreds of endpoints only stay maintainable if fetch/transform/score logic is shared and parameterized, with consistent JSON contracts and error handling. I'd mention timeouts, input validation and observability as first-class.

Likely follow-ups
  • How did you avoid duplicating logic across hundreds of APIs?
  • How did you version the APIs?
  • How did you handle a slow/failing data source?
Red-flag answer pattern
One giant script per endpoint with copy-pasted logic and no shared layers.
What to mention from the project
  • Hundreds of loop/equipment REST APIs
  • Shared fetch/transform/score layers
  • Structured JSON contracts
How to connect this to GenAI roles
🔗

The same layered design is exactly how you build a production RAG/agent API: a retrieval layer, an orchestration layer, and a thin API layer with consistent contracts — not model calls scattered through the code.

Q3How did you handle time-series data from Elasticsearch / databases?Advancedtechnical deep-diveSenior
Why the interviewer asks this

Time-series + Elasticsearch is messy; this checks real data-engineering experience, not toy datasets.

Model answer
  • Queried windows of plant sensor data per tag/loop from Elasticsearch/DB.
  • Handled gaps, irregular sampling, and out-of-range/garbage sensor values.
  • Pivoted long sensor streams into a wide, model-ready table aligned on timestamps.
  • Validated ranges/units before inference so bad data didn't silently corrupt scores.
Strong senior-level answer

I'd call out the failure modes that matter at plant scale: missing tags, clock/sampling misalignment, and sensor drift — and how validation gates protect downstream scores. I'd mention query efficiency (windowing, only pulling needed tags) for latency.

Likely follow-ups
  • How did you handle missing or irregular timestamps?
  • How did you keep Elasticsearch queries fast?
  • What did you do with out-of-range sensor values?
Red-flag answer pattern
Assuming clean, evenly-sampled data and ignoring validation.
What to mention from the project
  • Elasticsearch/DB time-series fetch
  • Gap handling, pivoting, range validation
  • Per-tag windowed queries
How to connect this to GenAI roles
🔗

This is the same discipline as document ingestion for RAG: fetch from a store, clean, validate quality, and shape into a model-ready form. 'Bad data in → bad scores out' maps to 'bad chunks in → hallucinations out'.

Q4Walk me through your data cleaning, pivoting, validation, and feature engineering.Advancedtechnical deep-diveSenior
Why the interviewer asks this

Feature engineering is where most production ML quality is won or lost; interviewers probe the details.

Model answer
  • Cleaning — drop/flag nulls, clip impossible values, de-duplicate, align units.
  • Pivoting — reshape long (tag, time, value) streams into a wide table per timestamp.
  • Validation — range/consistency checks; reject or flag windows that fail.
  • Features — rolling stats, rates of change, deviations from baseline/expected, lag features for KPIs.
Strong senior-level answer

I'd emphasize that validation is a gate, not an afterthought, and that features encoded domain meaning (deviation from expected operating range drives anomaly/health). I'd mention making transforms reusable and testable across loops.

Likely follow-ups
  • Which features mattered most for anomaly detection?
  • How did you test the transforms?
  • How did you encode domain knowledge?
Red-flag answer pattern
Vague 'I cleaned the data' with no specific steps or validation.
What to mention from the project
  • Pivot long→wide
  • Range/consistency validation gate
  • Rolling/deviation features
How to connect this to GenAI roles
🔗

Maps to chunking + metadata enrichment + retrieval preparation in RAG — both are 'shape raw source into the form the model needs, with quality gates'.

Q5How did the model inference and anomaly scoring work?Advancedtechnical deep-diveSenior
Why the interviewer asks this

Checks you understand serving models in production and the logic behind anomaly detection.

Model answer
  • Loaded model(s) once per service; ran inference on the prepared feature window.
  • Anomaly logic compared observed behavior against expected/normal operating ranges (model output and/or rule thresholds).
  • Produced a score per equipment/loop and returned it as structured JSON for the dashboard.
Strong senior-level answer

I'd be precise about anomaly = deviation from expected, and discuss thresholds and false-positive cost (operators ignore noisy alerts), plus how I kept inference latency acceptable for near-real-time dashboards.

Likely follow-ups
  • How did you set anomaly thresholds?
  • How did you handle false positives?
  • Model-based vs rule-based — what did you use where?
Red-flag answer pattern
Treating anomaly detection as a black box with no notion of thresholds or false-positive cost.
What to mention from the project
  • Per-equipment anomaly score
  • Expected-range deviation logic
  • Latency-aware inference
How to connect this to GenAI roles
🔗

Anomaly/quality scoring maps to LLM evaluation: a faithfulness/quality score is conceptually the same as a health/anomaly score — a number that flags when output deviates from 'good', with the same false-positive tradeoffs.

Q6Explain the health score, trip score, instrumentation score, and KPI calculation logic.Advancedtechnical deep-diveSenior
Why the interviewer asks this

Tests depth on the domain-specific outputs you built — generic answers reveal you didn't own them.

Model answer
  • Health score — aggregate condition of equipment from multiple signals/sub-scores.
  • Trip score — likelihood/severity related to a trip (unplanned shutdown) condition.
  • Instrumentation score — reliability/validity of the sensors/instruments feeding the loop.
  • KPIs — calculated process metrics (efficiency, deviations) from the cleaned time-series.

Each was computed from validated features and exposed via its own API field.

Strong senior-level answer

I'd explain how the scores compose (instrumentation quality gates trust in the health/trip scores — bad sensors shouldn't trigger false trips), and how I made the logic auditable so operators trusted it.

Likely follow-ups
  • How do these scores relate to each other?
  • What if instrumentation is unreliable — does health score still apply?
  • How did you validate the scores were trustworthy?
Red-flag answer pattern
Reciting names without explaining what each measures or how they combine.
What to mention from the project
  • Composite health from sub-scores
  • Instrumentation score gates trust
  • KPIs from validated features
How to connect this to GenAI roles
🔗

Composing sub-scores into a trustworthy overall score is exactly LLM-eval design: faithfulness + relevance + safety combine into an overall quality gate, and you must guard against one bad signal poisoning the verdict.

Q7How did you design the REST APIs and JSON response structure?Intermediatesystem designSenior
Why the interviewer asks this

Dashboard integration lives or dies on clean, consistent API contracts.

Model answer
  • Consistent endpoint pattern per loop/equipment.
  • Structured JSON: identifiers, timestamp/window, the scores (health/anomaly/trip/instrumentation), KPIs, and status/error fields.
  • Stable schema so the dashboard could render without special-casing.
  • Validation and clear error responses for bad/missing data.
Strong senior-level answer

I'd stress contract stability and predictability — the dashboard team depends on it, so a consistent schema and explicit error/empty states matter more than cleverness. I'd mention versioning and documenting the contract.

Likely follow-ups
  • How did you keep the schema consistent across hundreds of APIs?
  • How did you signal 'data unavailable' vs 'score is low'?
  • How did you version responses?
Red-flag answer pattern
Inconsistent ad-hoc JSON per endpoint that the frontend must special-case.
What to mention from the project
  • Consistent JSON contract
  • Explicit status/error fields
  • Stable schema for the dashboard
How to connect this to GenAI roles
🔗

Identical to designing a GenAI API: a structured response (answer, citations, confidence/quality, status) with a stable contract the frontend can rely on — and explicit handling for 'no answer / low confidence'.

Q8How did you deploy and debug the services on Kubernetes?AdvanceddeploymentSenior
Why the interviewer asks this

Separates engineers who shipped to production from those who only ran code locally.

Model answer
  • Packaged APIs as Docker images; deployed on Kubernetes.
  • Environment-specific config (data source endpoints, credentials) via config/secrets.
  • Health/readiness checks so traffic only hit ready pods.
  • Debugging via pod logs, checking config/connectivity to the data source, and reproducing failing requests.
Strong senior-level answer

I'd give a concrete debugging story: e.g. a service failing readiness due to a data-source connectivity/config issue, traced via logs and fixed in config — and mention resource limits, restarts, and rollout safety.

Likely follow-ups
  • Describe one production bug you debugged.
  • Readiness vs liveness probes — what's the difference?
  • How did you manage secrets/config across environments?
Red-flag answer pattern
Only knowing 'it ran in a container' with no health checks, logs, or a real debugging example.
What to mention from the project
  • Docker images + K8s deploy
  • Health/readiness checks
  • A concrete log-based debugging story
How to connect this to GenAI roles
🔗

Production GenAI apps deploy the same way — containerized FastAPI services on Kubernetes with health checks, config/secrets, and log-based debugging. This experience transfers 1:1.

Q9How did you integrate the APIs with the TCG Mcube dashboard?Intermediatesystem designSenior
Why the interviewer asks this

Enterprise integration is a real differentiator — it shows you delivered something users actually used.

Model answer
  • The Mcube application consumed the backend APIs and rendered health/anomaly/KPI views per loop/equipment.
  • Agreed a stable JSON contract with the dashboard/frontend side.
  • Did integration testing to confirm the dashboard rendered scores and trends correctly.
  • Coordinated on error/empty states so the UI degraded gracefully.
Strong senior-level answer

I'd emphasize the contract + cross-team coordination and that real users (plant operators) depended on it, so reliability and clear empty/error states mattered. I'd mention how integration issues were caught and resolved.

Likely follow-ups
  • How did you coordinate the API contract with the dashboard team?
  • What happened when an API returned no data?
  • How did you test the integration?
Red-flag answer pattern
Treating integration as 'I returned JSON and someone else's problem' with no contract or testing.
What to mention from the project
  • Mcube consumed the APIs
  • Agreed JSON contract
  • Integration testing + graceful empty states
How to connect this to GenAI roles
🔗

This is enterprise GenAI frontend integration: a RAG/agent backend exposing a contract that an internal app/dashboard consumes, with the same need for stable contracts and graceful degradation.

Q10How does this production ML project map to GenAI systems (RAG, agents, LLMOps, monitoring)?SeniorbehavioralSenior/Staff
Why the interviewer asks this

For a GenAI role with a non-GenAI project, this is the decisive question — can you bridge it credibly without overclaiming?

Model answer

Be explicit and honest: it wasn't a GenAI project, but the production engineering is the same:

  • ML APIs → LLM/RAG/agent APIs.
  • Elasticsearch data fetch → document/vector retrieval.
  • Feature engineering → chunking + metadata enrichment.
  • Health/anomaly scores → eval/quality/hallucination scores.
  • Equipment dashboard → GenAI observability dashboard.
  • Kubernetes deploy + debugging → production GenAI deployment + RAG/agent debugging.
Strong senior-level answer

I'd say production GenAI is mostly the engineering around the model — pipelines, serving, observability, deployment, integration — which is exactly what I did here; the new part I'd add is retrieval quality, prompting and LLM-specific evaluation, which I've been building hands-on.

Likely follow-ups
  • What's genuinely new for you in GenAI?
  • Would you call this a GenAI project? (Say no, honestly.)
  • How would you turn this into a RAG system?
Red-flag answer pattern
Overclaiming the project used GenAI/LLMs when it didn't — instant credibility loss.

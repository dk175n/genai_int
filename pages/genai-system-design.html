GenAI System Design

The reusable framework for any GenAI system-design interview — clarify, requirements, architecture, ingestion, retrieval, orchestration, safety, evaluation, observability, then tradeoffs and failure modes. Learn it once here; apply it on the seven worked designs.

Why this topic shows up in interviews

For 5–10 yr engineers, the design round often decides the level (and offer). GenAI design adds probabilistic behavior, retrieval quality, hallucination, cost/latency of model calls, safety and evaluation on top of classic concerns — and interviewers want to see you handle all of it without prompting.

Interview answers — 60-second & 3-minute

60 sec The crisp interview answer

“I drive GenAI design with one framework: clarify the product and constraints; set functional and non-functional requirements (quality, latency, cost, safety); draw a layered architecture — ingestion, retrieval, model/orchestration, safety, evaluation, observability; then I lead with tradeoffs and failure modes. The LLM is one component inside a real system that handles auth, retrieval, validation, monitoring, cost control and feedback — saying that up front sets the level.”

3 min The deeper answer that signals seniority

I open by clarifying: who are the users, what's the data, what's the quality/latency/cost/safety bar, what scale? Then I state functional requirements (what it does) and non-functional (p95 latency, cost/query, accuracy/groundedness target, privacy/compliance).

Then the layered architecture: ingestion (load, chunk, embed, index, refresh), retrieval (hybrid + rerank, access control), orchestration (prompts, model routing, agents/tools), safety (input/output guards, injection defense), evaluation (golden set, online sampling), observability (tracing, cost/latency/quality dashboards). I make the LLM clearly one box among many.

Finally — and this is the seniority differentiator — I lead with tradeoffs (RAG vs fine-tune, model size vs cost/latency, caching vs freshness), failure modes (bad retrieval, hallucination, provider outage, injection) and how I'd detect and mitigate each. I keep checking back with the interviewer and adjust to their priorities.

Core concept

The universal framework

  1. Clarify product, users, use cases
  2. Quality / safety / latency / cost goals
  3. High-level architecture
  4. Data & knowledge ingestion
  5. Retrieval / memory / tools
  6. Prompt & model orchestration (routing)
  7. Output validation & guardrails
  8. Evaluation & feedback loops
  9. Observability, reliability, scaling, deployment
  10. Tradeoffs
  11. Failure modes

QDesign a GenAI assistant over our internal documents. Walk me through it.Seniorsystem designSenior/Stafffreq: highBoth
Model answer

Clarify users/data/scale/quality bar. Then: ingestion (chunk, embed, index, refresh, ACLs), hybrid retrieval + rerank with access control, grounded generation with citations and abstain, output guardrails, evaluation (golden set + online sampling), observability (tracing, cost/latency/quality). Close with tradeoffs and failure modes.

Strong senior-level answer

I'd front-load the non-functional bar (latency, cost/query, groundedness target, compliance) because it drives every choice, enforce access control at retrieval, and spend real time on evaluation and failure modes (bad retrieval, hallucination, injection, provider outage). I keep the LLM as one component and make the system observable and updatable.

Likely follow-ups
  • How do you keep the index fresh?
  • How do you handle 100 docs vs 100M?
  • How do you measure if it's good?
Red-flag answer pattern
'User asks the LLM and it answers' — no retrieval quality, safety or evaluation.

QHow is GenAI system design different from classic system design?Intermediatesystem designSeniorfreq: mediumBoth
Model answer

It adds probabilistic model behavior, retrieval quality, hallucination risk, prompt/version management, token cost and model latency, safety/injection, and evaluation/feedback loops — on top of the usual APIs, storage, caching and scaling.

Strong senior-level answer

The mental shift is that correctness is statistical, so evaluation and observability become first-class design components, not afterthoughts, and cost/latency are dominated by model calls. I design the feedback loop (production failures → eval set → improvement) into the architecture from the start.

Likely follow-ups
  • Where does evaluation live in the architecture?
Red-flag answer pattern
Treating it as identical to a CRUD service design.
QDesign a customer support RAG bot.Seniorsystem designSeniorfreq: highBoth
Model answer

Customer-facing RAG over a help center. Run the framework, and emphasize the differences from an internal assistant: external users (higher abuse/injection risk), brand risk from wrong answers, and a clean handoff to human agents.

Key pieces: hybrid retrieval over KB, a coverage gate that escalates when unsure, citations, strong input guards, ticket creation with full context, and a feedback loop into evaluation.

Strong senior-level answer

A senior optimizes for trust over raw deflection (a confident wrong answer is worse than an escalation) and measures escalation precision and CSAT, not just how many tickets were auto-handled.

Likely follow-ups
  • Answer vs escalate — how do you decide?
  • How do you prevent cross-customer leaks?
Red-flag answer pattern
Maximizing auto-answer rate with no abstain/escalation path.

QDesign a legal/finance document Q&A system.Seniorsystem designSenior/Stafffreq: mediumBoth
Model answer

High-stakes RAG where a confident wrong answer creates liability. Priorities flip toward precision and auditability:

  • Mandatory clause-level citations; never answer without a source.
  • Conservative abstain — guess nothing.
  • Version-aware ingestion (cite the exact document version).
  • A faithfulness gate that blocks unsupported claims, plus a full audit trail.
Strong senior-level answer

A senior accepts higher cost, latency, and abstain rate as the correct trade in a regulated domain, adds a verification pass and human-in-the-loop before any action, and references compliance/retention requirements.

Likely follow-ups
  • Would you ever answer without a citation?
  • How do you make it reproducible?
Red-flag answer pattern
Optimizing for coverage/UX where wrong answers carry legal risk.
Related hands-on task / project

Document Q&A (regulated) design.

QDesign a production RAG platform used by many teams.Seniorsystem designStafffreq: emergingBoth
Model answer

A shared platform, not one app: shared ingestion, retrieval and a model gateway (routing, caching, fallback, rate limits), a shared eval harness and guardrail layer every app inherits, per-tenant isolation, cost attribution, and central observability.

Strong senior-level answer

A senior makes standards executable — an app can't onboard without passing baseline eval and inheriting guardrails — designs for noisy-neighbor and cost attribution from day one, and keeps the gateway highly available.

Likely follow-ups
  • How do you attribute cost per tenant?
  • How do you isolate a shared cache?
Red-flag answer pattern
One shared index for all tenants with no isolation story.

Hands-on task

Portfolio connection

Two design docs in your portfolio (e.g. knowledge assistant + RAG platform) are excellent senior signals and great mock-interview fodder.

Common mistakes

  • Describing only the happy path
  • Forgetting evaluation/observability as design components
  • No access control / safety layer
  • Not clarifying requirements before drawing boxes

Revision checklist

  • I can run the 11-step framework from memory
  • I lead with tradeoffs + failure modes
  • I treat eval & observability as architecture
  • I time-box a 10-minute explanation

Core Track

Agentic AI

When an LLM stops just answering and starts doing — calling tools, planning multi-step work, and maintaining state. Covers tool/function calling, ReAct, planner-executor, LangGraph-style stateful agents, agent memory, and the guardrails that keep them safe.

Flagship topicRising fast
AudienceEngineers building or explaining tool-using agents.
GoalDesign an agent, choose the right control pattern, and reason about reliability and safety.
You'll learn
  • The agent loop: perceive → decide → act → observe
  • Tool/function calling done reliably
  • ReAct vs planner-executor vs stateful graphs
  • Agent memory (short-term, long-term, episodic)
  • Why agents fail and how to guardrail them
  • When NOT to use an agent (workflow vs agent)

Why this topic shows up in interviews

Agents are the current frontier of hiring demand. Interviewers want to know you can build one that's reliable, not a flashy demo that loops forever or calls the wrong tool — so they probe control flow, failure handling and guardrails.

Interview answers — 60-second & 3-minute

60 sec The crisp interview answer

“An agent is an LLM in a loop with tools: it decides which tool to call, the runtime executes it, the result goes back into context, and it repeats until done. Tool calling is the mechanism; ReAct (reason+act) and planner-executor are control patterns; LangGraph models the agent as a stateful graph with checkpoints and human-in-the-loop. The hard part is reliability — bounded loops, validated tool calls, and guardrails — and knowing when a deterministic workflow beats an agent.”

3 min The deeper answer that signals seniority

Mechanism first: the model is given tool schemas; it emits a structured tool call; the runtime executes the real function (DB query, API call, search); the result is appended to context; loop until the model emits a final answer. That loop is the agent.

Control patterns: ReAct interleaves reasoning and acting step-by-step — flexible, good for open-ended tasks, but can wander. Planner-executor makes a plan up front then executes steps — more predictable, easier to debug. Stateful graphs (LangGraph) model nodes/edges with explicit state, checkpointing and human-in-the-loop — the production-grade option when you need control, retries and observability.

Reliability is where senior signal lives: bound the loop (max steps), validate every tool call against a schema, make tools idempotent and least-privilege, add timeouts/retries, and put guardrails on both tool inputs and final outputs. And I always ask whether an agent is even needed — if the steps are known, a deterministic workflow is cheaper, faster and safer.

Agent memory

  • Short-term: the conversation/scratchpad in the context window.
  • Long-term: facts persisted across sessions (often a vector store) — retrieved like RAG.
  • Episodic / state: the structured state the graph carries between steps (checkpointed).
⚠️
Workflow vs agent
If you know the steps in advance, build a deterministic workflow (cheaper, faster, testable). Reach for an agent only when the path is genuinely dynamic. 'We used an agent because it's cool' is a red flag; 'we used a workflow because the steps were fixed' is senior judgment.

QHow does an agent actually work — explain tool/function calling.Intermediatetechnical deep-diveSeniorfreq: highBoth
Model answer

You give the model tool schemas. It emits a structured tool call; your runtime executes the real function and returns the result into context; the model decides the next action or finishes. The model never runs code itself — your runtime does.

Strong senior-level answer

I stress that the model only proposes calls — the runtime is the trust boundary, so I validate arguments against the schema, enforce least privilege and idempotency, add timeouts/retries, and bound the loop. That framing (model proposes, runtime disposes) is what makes agents safe.

Likely follow-ups
  • What stops it from looping forever?
  • How do you handle a tool that fails?
  • How do you prevent it calling a destructive tool wrongly?
Red-flag answer pattern
Thinking the LLM executes the tool/code itself.
Related hands-on task / project

Build a 2-tool agent with a bounded loop and input validation (see Build First Agent App).

QReAct vs planner-executor — when do you use each?Advancedtechnical deep-diveSeniorfreq: mediumBoth
Model answer

ReAct for open-ended, adaptive tasks where steps emerge from observations. Planner-executor when the task decomposes cleanly up front and you want predictability and easier debugging.

Strong senior-level answer

In practice I lean planner-executor or an explicit graph for production because predictability, cost control and observability matter more than raw flexibility. Pure ReAct is great for exploration but its variable step count makes cost/latency and testing harder. A hybrid — plan, then ReAct within steps — is common.

Likely follow-ups
  • How do you bound cost in ReAct?
  • How do you test a planner-executor agent?
Red-flag answer pattern
Treating ReAct as the only/always-best pattern.
QWhy do agents fail in production and how do you make them reliable?Advancedtechnical deep-diveSeniorfreq: highBoth
Model answer

Common failures: infinite/looping behavior, wrong tool or wrong arguments, cascading errors from a bad early step, cost blowups, and prompt injection via tool outputs. Fixes: bounded loops, schema-validated tool calls, idempotent least-privilege tools, retries/timeouts, and guardrails on inputs and outputs.

Strong senior-level answer

I add explicit state and checkpointing (so I can retry a step, not the whole run), per-step and per-run budgets, human-in-the-loop on high-risk actions, and full tracing so I can see the decision path. I also treat tool outputs as untrusted input (injection surface). Reliability is engineering around the loop, not a better prompt.

Likely follow-ups
  • How do you trace an agent run?
  • Where do you put human-in-the-loop?
  • How is tool output an injection vector?
Red-flag answer pattern
No mention of bounding the loop, budgets, or treating tool output as untrusted.
QWhat kinds of memory does an agent need?Intermediatetechnical deep-diveMid–Seniorfreq: mediumBoth
Model answer

Short-term (the in-context conversation/scratchpad), long-term (facts persisted across sessions, retrieved like RAG), and structured state the agent carries between steps (checkpointed in a graph).

Strong senior-level answer

I'm deliberate about what's worth persisting — over-stuffing long-term memory adds noise and cost. Long-term memory is just RAG over the agent's own history with the same retrieval-quality concerns; structured state is what lets me checkpoint and resume. I separate 'what the agent knows' from 'where it is in the task'.

Likely follow-ups
  • How do you avoid memory becoming noise?
  • How do you checkpoint state?
Red-flag answer pattern
Treating 'memory' as one vague blob.
QWhen would you NOT build an agent?Intermediatesystem designSeniorfreq: mediumBoth
Model answer

When the steps are known in advance — then a deterministic workflow (with maybe one LLM call per step) is cheaper, faster, more testable and more reliable than an autonomous loop.

Strong senior-level answer

I default to the least-agentic thing that works. Most 'agent' problems are really pipelines with one or two LLM decisions. I escalate to a full agent only when branching is genuinely data-dependent at runtime. Showing that restraint signals production maturity.

Likely follow-ups
  • Give an example you'd build as a workflow not an agent.
Red-flag answer pattern
Defaulting to agents for everything.

QHow do you prevent an agent from looping forever?Intermediatetechnical deep-diveSeniorfreq: highBoth
Model answer
  • Hard cap on steps (max iterations) — stop and return a best-effort or escalate when hit.
  • Per-run token/cost budget.
  • Detect no-progress loops (same tool + same args repeated) and break.
  • Timeouts on each tool call.
Strong senior-level answer

A senior treats the bound as a product decision (what happens when we hit the cap — degrade, escalate, or fail honestly?) and logs how often runs hit the limit as a signal that the task or tools need redesign.

Likely follow-ups
  • What do you return when the cap is hit?
  • How do you detect a no-progress loop?
Red-flag answer pattern
Relying on the model to 'know when to stop' with no hard limit.
QHow do you secure an agent's tools?Advancedsystem designSeniorfreq: highBoth
Model answer

Safety comes from what the tools can do, not from the prompt:

  • Least privilege — each tool gets the minimum capability (e.g. read-only DB, send-mail only to allow-listed addresses).
  • Validate every tool argument against a schema before executing.
  • Make tools idempotent; add timeouts and rate limits.
  • Require human approval for destructive/irreversible actions.
  • Treat tool outputs as untrusted input (indirect injection).
Strong senior-level answer

A senior assumes the model can be manipulated and designs so even a hijacked agent can't do damage — plus audit-logs every action with the reasoning that triggered it.

Likely follow-ups
  • How do you make a SQL tool safe?
  • Where do you put human-in-the-loop?
Red-flag answer pattern
Giving the agent broad write/delete access and trusting it to behave.

QHow do you evaluate an agent?Advancedtechnical deep-diveSeniorfreq: mediumBoth
Model answer

Define task cases with expected outcomes and measure:

  • Task success rate (did it achieve the goal / produce the right final answer?).
  • Correct tool selection and argument accuracy.
  • Average steps and cost per task (efficiency).
  • Tool-error and recovery rate.

Include adversarial cases (ambiguous goals, injection via tool output).

Strong senior-level answer

A senior evaluates the trajectory, not just the final answer — checking the agent took a sensible path — and runs the suite as CI so reliability doesn't regress.

Likely follow-ups
  • How do you evaluate the path, not just the result?
  • How big is your task set?
Red-flag answer pattern
Only checking the final answer 'looked right' on a couple of runs.
Related hands-on task / project

Add a task-success eval to your agent app.

QHow does LangGraph help with stateful agents?Advancedtechnical deep-diveSeniorfreq: emergingBoth
Model answer

LangGraph models the agent as a graph of nodes/edges with explicit shared state. That gives you:

  • Explicit state passed between steps (not buried in the prompt).
  • Conditional routing — next node depends on results.
  • Checkpointing — retry/resume a single step, not the whole run.
  • Human-in-the-loop nodes for approval.
Strong senior-level answer

A senior values it for control and observability but notes the same can be done in disciplined plain Python — the graph is a convenience for managing state and retries, not magic.

Likely follow-ups
  • What does the checkpointer give you operationally?
  • When is plain code better?
Red-flag answer pattern
Using it because it's popular, unable to say what problem it solves.

QWhat are common multi-agent failure modes?Advancedsystem designSenior/Stafffreq: emergingBoth
Model answer
  • Coordination chaos / agents looping messages between each other.
  • Error cascade — one agent's mistake corrupts everything downstream.
  • Cost blowup from many agents each making many calls.
  • Diffused responsibility — no clear owner of the final output.
  • Context loss when passing work between agents.
Strong senior-level answer

A senior first questions whether multi-agent is even justified (vs one agent or a workflow), then mitigates with a supervisor, bounded total steps, checkpointed shared state, and tracing across agents.

Likely follow-ups
  • When is multi-agent actually worth it?
  • How do you stop inter-agent loops?
Red-flag answer pattern
Proposing many agents because it sounds sophisticated, with no coordination or cost story.

Common mistakes

  • Trusting tool outputs (injection)
  • Over-privileged tools
  • Using an agent where a workflow suffices

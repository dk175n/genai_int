LLM Evaluation

How you prove a GenAI feature works and keep it working. Golden datasets, LLM-as-judge, the metrics that matter (faithfulness, relevance, correctness), and treating evaluation as CI — the discipline that separates engineers from prompt tinkerers.

Senior signalHigh frequency
AudienceEngineers who must ship and maintain LLM features responsibly.
GoalDesign an evaluation harness and defend your quality numbers in an interview.
You'll learn
  • Why eval is hard for non-deterministic systems
  • Golden datasets: building and sizing them
  • LLM-as-judge and how to trust it
  • Core metrics: faithfulness, relevance, correctness
  • Offline vs online evaluation
  • Eval as CI / regression gating

Why this topic shows up in interviews

Anyone can make a demo look good once. Interviewers ask 'how do you know it works?' to find engineers who think in measurement. A strong eval answer is one of the clearest seniority signals in the loop.

Interview answers — 60-second & 3-minute

60 sec The crisp interview answer

“Evaluation is how I keep a non-deterministic system trustworthy. I build a small golden dataset of inputs with expected outputs/criteria, score outputs with a mix of deterministic checks and LLM-as-judge (faithfulness, relevance, correctness), and run it like CI on every prompt/model/retrieval change. In production I sample live traffic and run the same judges, plus capture user feedback — so quality is a number I can track, not a vibe.”

3 min The deeper answer that signals seniority

Two surfaces: offline (a curated golden set run pre-deploy, like a test suite) and online (sampled production traffic scored continuously). For RAG I split retrieval metrics (recall@k, MRR) from generation metrics (faithfulness, answer relevance, correctness).

LLM-as-judge scales scoring but must be trusted: I use a clear rubric, low temperature, pairwise or reference-based scoring where possible, and I calibrate the judge against a sample of human labels (and watch for position/verbosity bias). I version the judge prompt like any other.

The discipline that matters most: eval as CI. Every prompt edit, model swap or retrieval change runs the golden set and must not regress. That's what lets a team iterate on LLM features without fear, and it's exactly what interviewers are listening for.

Golden dataset

30–200 representative cases is enough to start — coverage matters more than size. Include happy paths, edge cases, adversarial inputs, and 'should abstain' cases. Grow it from real failures (every production bug becomes a test case).

cases = load_golden("eval/golden.jsonl")
scores = [judge(faithfulness, run_app(c.input), c) for c in cases]
mean = sum(scores) / len(scores)

Every production failure becomes a test
The fastest way to a strong golden set is to add each real-world miss as a case. Your eval set then encodes exactly the mistakes you've already made — and guarantees you don't repeat them.

QHow do you evaluate a GenAI feature before and after shipping it?Advancedtechnical deep-diveSeniorfreq: highBoth
Model answer

Offline golden set with judges as a pre-deploy gate; online sampling of live traffic scored by the same judges plus user-feedback signals. Split metrics by stage (retrieval vs generation for RAG).

Strong senior-level answer

I run eval as CI so no change ships without passing the golden set, and I close the loop by turning production failures into new cases. I calibrate LLM judges against human labels and track the judge's agreement over time. The goal is that 'is it better?' is always answerable with a number.

Likely follow-ups
  • How big is your golden set?
  • How do you trust the judge?
  • Offline passes but users complain — what now?
Red-flag answer pattern
'We manually check a few outputs.'
Related hands-on task / project

Add an offline eval CI gate + online sampling to your RAG app.

QHow do you trust LLM-as-judge?Advancedtechnical deep-diveSeniorfreq: mediumBoth
Model answer

Use a clear rubric, low temperature, reference-based or pairwise comparisons, and calibrate against a human-labeled sample. Watch for position, verbosity and self-preference bias.

Strong senior-level answer

I measure judge–human agreement (e.g. on 50 labeled cases) and only trust the judge where agreement is high; for high-stakes calls I keep humans in the loop. I also use a stronger/different model as judge than the one being judged to reduce self-preference, and I version the judge prompt and re-validate it when I change it.

Likely follow-ups
  • What biases affect LLM judges?
  • Same model as judge and generator — problem?
Red-flag answer pattern
Treating judge scores as ground truth with no calibration.
QWhat's a golden dataset and how do you build one?Intermediatetechnical deep-diveMid–Seniorfreq: highBoth
Model answer

A curated set of representative inputs with expected outputs or pass/fail criteria, used to measure quality and catch regressions. Build from real queries, edge cases, adversarial inputs, and abstain cases; grow it from production failures.

Strong senior-level answer

I prioritize coverage over size — 50 well-chosen cases beat 5,000 near-duplicates. I tag cases by category so I can see which kind of query regressed, and I keep an 'adversarial' slice (injections, out-of-scope) as a permanent gate.

Likely follow-ups
  • How do you keep it from going stale?
Red-flag answer pattern
Thinking you need thousands of cases before you can start.
QOffline vs online evaluation — what's the difference and why both?Advancedtechnical deep-diveSeniorfreq: highBoth
Model answer
  • Offline — run a curated golden set before deploy, like a test suite. Gates releases; catches regressions early.
  • Online — score a sample of real production traffic plus user feedback. Catches what your golden set didn't anticipate.

You need both: offline can't cover the real distribution; online catches issues after users hit them.

Strong senior-level answer

A senior runs the same judges in both, so offline and online share one quality definition, and feeds online failures back into the offline golden set — a closed loop.

Likely follow-ups
  • How do you score online traffic without labels?
  • How do you detect input drift?
Red-flag answer pattern
Relying only on offline (misses real usage) or only on online (catches regressions after users do).

RAGAS — RAG evaluation metrics

QHow do you evaluate retrieval separately from generation in RAG?Advancedtechnical deep-diveSeniorfreq: highBoth
Model answer
  • Retrieval — on a labeled set of question→relevant-doc, measure recall@k, MRR, nDCG. Did the right chunks come back?
  • Generation — given the retrieved context, measure faithfulness (claims supported), answer relevance, and correctness vs a golden answer.

Splitting them tells you which half to fix when quality drops.

Strong senior-level answer

A senior insists on this split because a single blended score hides whether a regression is retrieval or generation — the two have completely different fixes (chunking/hybrid vs prompt/grounding).

Likely follow-ups
  • What if retrieval is good but the answer is wrong?
  • How do you build the labeled retrieval set?
Red-flag answer pattern
One combined 'quality' number with no way to localize a failure.

RAGAS — RAG evaluation metrics

Hands-on task

Build a 40-case golden set for your RAG app (tagged by category, with adversarial cases), wire faithfulness + recall@k judges, and gate a CI run on no-regression.

Portfolio connection

An eval harness with published numbers is one of the strongest portfolio signals. Add an 'Evaluation' section to your RAG app README and screenshots to the monitoring dashboard.

Common mistakes

  • Eyeballing outputs
  • Trusting LLM judges without calibration
  • One blended metric that hides retrieval vs generation
  • Never updating the golden set

Revision checklist

  • I can design offline + online eval
  • I can build and size a golden set
  • I can calibrate an LLM judge
  • I run eval as CI

Guardrails & Security

The rules an answer must pass before it ships, and the attacks you must assume. Input/output validation, prompt-injection defense, PII handling, and tool/agent security — increasingly a dedicated interview round as companies move GenAI to production.

Dedicated roundEnterprise must-have
AudienceEngineers shipping GenAI to real users / enterprise.
GoalDesign layered guardrails and reason about prompt injection and agent security.
You'll learn
  • The guardrail layers: input, retrieval, output, action
  • Prompt injection (direct & indirect) and defenses
  • PII and data-leak prevention
  • Tool/agent security: least privilege, human-in-the-loop
  • Jailbreaks and content safety
  • Why guardrails ≠ a single system prompt line

Why this topic shows up in interviews

As GenAI hits production, safety incidents (leaked data, injected instructions, destructive agent actions) are board-level risks. Interviewers test whether you treat security as architecture, not a prompt afterthought.

Interview answers — 60-second & 3-minute

60 sec The crisp interview answer

“Guardrails are layered checks around the model, not a line in the prompt. On input: validate, detect injection, strip/redact PII, enforce scope. On retrieval: access control so users only see permitted docs. On output: validate format, check groundedness, block PII/unsafe content, enforce policy. On actions: least-privilege tools, human-in-the-loop for high-risk operations. Prompt injection — especially indirect, via retrieved or tool content — is the signature GenAI threat, so I treat all external/tool content as untrusted.”

3 min The deeper answer that signals seniority

I think in four layers. Input: schema validation, scope/topic check, PII redaction, and injection detection. Retrieval: enforce per-user access control on the corpus (a huge real-world leak vector — never retrieve documents the user can't see). Output: structured validation, groundedness/faithfulness check, PII and unsafe-content filters, and policy enforcement before the response leaves. Action: tools are least-privilege and idempotent, destructive actions require confirmation or human-in-the-loop.

Prompt injection is the core threat. Direct: the user tries to override instructions. Indirect: malicious instructions hidden in a web page, document or tool output that the model then follows — this is the dangerous one for RAG and agents. Defenses: treat all retrieved/tool content as data not instructions, separate trusted system instructions from untrusted content, constrain tool permissions, validate outputs, and never let model output alone trigger an irreversible action.

There's no single fix — it's defense in depth plus monitoring and red-teaming. I'd add injection/jailbreak cases to the eval set as a permanent gate.

Indirect injection is the one to name
Most candidates only mention 'ignore previous instructions'. Naming indirect injection — malicious instructions inside retrieved documents or tool outputs that a RAG/agent system then executes — is the senior signal. The fix is architectural: untrusted content is data, never instructions, and tools are least-privilege.

QWhat is prompt injection and how do you defend against it?Advancedtechnical deep-diveSeniorfreq: highBoth
Model answer

An attack where malicious instructions in user input (direct) or in retrieved/tool content (indirect) make the model ignore its real instructions. Defend by separating trusted instructions from untrusted content, treating retrieved/tool data as data not commands, validating outputs, and constraining tool permissions. Defense in depth, not one trick.

Strong senior-level answer

I emphasize indirect injection as the dangerous case for RAG/agents — a poisoned document can hijack the system. Architecturally: untrusted content never carries authority, tools are least-privilege and idempotent, irreversible actions need confirmation/HITL, and I add injection cases to the eval gate and monitor for anomalies. There's no silver bullet, so I assume some attempts succeed and limit blast radius.

Likely follow-ups
  • How would you exfiltrate data from a naive RAG agent?
  • Direct vs indirect — which is worse for agents?
  • How do you test for injection?
Red-flag answer pattern
'Just tell the model in the system prompt to ignore injection attempts.'
Related hands-on task / project

Add an injection test suite to your RAG/agent app and an output guard.

QHow do you stop a RAG system from leaking data a user shouldn't see?Advancedsystem designSeniorfreq: highBoth
Model answer

Enforce access control at retrieval: filter the corpus by the user's permissions before/while retrieving, so restricted documents are never even candidates. Don't rely on the model to 'not mention' restricted content.

Strong senior-level answer

Access control belongs in the retrieval layer with metadata-based filters tied to the user's identity/roles, ideally enforced in the vector store query itself, plus an output PII/secret check as a backstop. I also segregate indexes for hard tenancy boundaries. The model is the last place I'd trust to enforce permissions.

Likely follow-ups
  • Per-document ACLs vs separate indexes — when each?
  • How do you handle a shared multi-tenant index?
Red-flag answer pattern
Relying on a prompt instruction to keep restricted data hidden.
QHow do you secure an agent that can take actions (send email, run SQL)?Advancedsystem designSeniorfreq: mediumBoth
Model answer

Least-privilege, idempotent tools; validate every tool argument; require confirmation/human-in-the-loop for destructive or irreversible actions; egress controls; full audit logging; and treat tool outputs as untrusted (injection).

Strong senior-level answer

I scope each tool to the minimum capability (read-only SQL with row limits, send-email only to internal/allow-listed addresses), put irreversible actions behind explicit approval, and log every action with the reasoning that triggered it for audit. I assume the model can be manipulated, so safety comes from what the tools can't do, not from the model behaving.

Likely follow-ups
  • Where exactly do you put the human-in-the-loop?
  • How do you make a SQL tool safe?
Red-flag answer pattern
Giving an agent broad write/delete access and trusting it to behave.

OWASP Top 10 for LLM Apps

QHow do you handle PII in a GenAI app?Advancedsystem designSeniorfreq: highBoth
Model answer
  • Input — detect and redact/tokenize PII before it reaches the model or logs.
  • Retrieval — access control so PII docs only reach permitted users.
  • Output — scan responses for PII/secret leakage before returning.
  • Logs/traces — redact PII so observability doesn't become a leak.
  • Respect data-residency and retention rules.
Strong senior-level answer

A senior balances redaction against utility (over-redaction breaks answers), keeps an audit trail, and treats traces/caches as PII surfaces people forget about.

Likely follow-ups
  • Doesn't redaction remove needed context?
  • How do you handle PII in cached responses?
Red-flag answer pattern
Sending raw PII to a third-party model and logging it in plaintext.

NIST AI Risk Management Framework

QWhat guardrails should an enterprise GenAI app have?Seniorsystem designSenior/Stafffreq: highBoth
Model answer

A layered set, not one check:

  • Input — validation, scope/topic check, PII redaction, injection detection.
  • Retrieval — per-user access control.
  • Output — groundedness check, PII/secret filter, policy/brand compliance, citation presence.
  • Action — least-privilege tools, human-in-the-loop on irreversible actions.
  • Cross-cutting — audit logging, eval gate including adversarial cases, monitoring.
Strong senior-level answer

A senior maps these to a recognized framework (OWASP LLM Top 10 / NIST AI RMF) so the controls are defensible to security and compliance, and makes them enforced platform-wide rather than per-app.

Likely follow-ups
  • How do you enforce these across many apps?
  • Which framework do you reference?
Red-flag answer pattern
Treating guardrails as a single system-prompt instruction.

  • Forgetting indirect injection
  • Enforcing permissions in the prompt instead of retrieval
  • Over-privileged agent tools

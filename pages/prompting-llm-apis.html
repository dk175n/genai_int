Core Track

Prompting & LLM APIs

The practitioner layer: how to talk to an LLM API like an engineer — system vs user roles, structured output, function/tool calling, streaming, and the prompt patterns that actually move quality (not prompt-magic folklore).

IntermediateCoding round
AudienceEngineers who'll be building LLM features behind an API.
GoalDesign reliable prompts and integrate LLM APIs with structured output, streaming and tool calling.
You'll learn
  • Roles: system / user / assistant and how to use them
  • Prompt patterns that work: instruction + context + examples
  • Structured / JSON output and why schemas beat parsing
  • Function/tool calling basics
  • Streaming and its UX/latency tradeoffs

Why this topic shows up in interviews

Every applied loop probes whether you can get reliable, parseable, grounded output from a model — not whether you know clever prompt tricks. Structured output and tool calling come up constantly in coding rounds.

Interview answers — 60-second & 3-minute

60 sec The crisp interview answer

“I structure prompts as role + clear instruction + context + (optional) few-shot examples + an explicit output contract. For anything a program consumes I force structured output via a schema/tool-call rather than parsing free text. I stream when the user is waiting on long output, and I keep prompts versioned because they're config, not code comments.”

3 min The deeper answer that signals seniority

I think in terms of an output contract. The system prompt sets role and rules; the user message carries the task and context; for machine-consumed output I use the provider's structured-output / tool-calling so the model returns validated JSON instead of prose I have to regex.

Prompt quality levers that actually matter: be explicit about the task and constraints, put the most important instructions early and late, provide 1–3 high-quality few-shot examples for format adherence, and ask for 'I don't know' when context is insufficient. I treat prompts as versioned config so I can A/B and roll back — never as buried string literals.

Core concept

The output contract

Decide first: is a human or a program reading this? Programs get schemas; humans get prose. Don't parse free text when the API can return validated JSON.

python
from pydantic import BaseModel
from anthropic import Anthropic

class Triage(BaseModel):
    category: str
    urgency: int  # 1-5
    needs_human: bool

client = Anthropic()
# Tool calling = the reliable way to get structured output
resp = client.messages.create(
    model="claude-opus-4-8", max_tokens=300,
    tools=[{"name": "emit_triage", "description": "Return the triage decision",
            "input_schema": Triage.model_json_schema()}],
    tool_choice={"type": "tool", "name": "emit_triage"},
    messages=[{"role": "user", "content": email_text}],
)
decision = Triage(**resp.content[0].input)  # validated, no regex

Interview questions on this topic (5)

Click any question to reveal a model answer, a stronger senior-level answer, likely follow-ups and the red-flag pattern interviewers listen for.

QHow do you get reliable structured (JSON) output from an LLM?IntermediatecodingSeniorfreq: highBoth
Model answer

Use the provider's structured-output / function-calling with a JSON schema (e.g. Pydantic), set tool_choice to force the tool, and validate the result. On validation failure, retry with the error fed back. Don't regex free text.

Strong senior-level answer

I define the schema as the single source of truth (Pydantic model → json schema → validation), force the tool call, and add a bounded repair loop that returns the validation error to the model. I also keep temperature low and log schema-failure rate as a quality metric — a rising rate usually means a prompt or schema drift, not a model problem.

Likely follow-ups
  • What do you do on a validation failure?
  • How does this interact with streaming?
Red-flag answer pattern
Parsing JSON out of prose with string operations and no validation.
Related hands-on task / project

Build the triage extractor above; add a repair loop and log failure rate.

QWhen do you use streaming, and what does it complicate?Intermediatetechnical deep-diveSeniorfreq: mediumBoth
Model answer

Stream when a human waits on long output — it cuts perceived latency. It complicates output validation/guardrails (you may emit text before you've seen it all) and makes structured parsing harder.

Strong senior-level answer

I stream user-facing prose, but for machine-consumed or safety-critical output I either don't stream or buffer-and-validate before release. A common pattern: stream tokens to the UI but run guardrail checks on the completed message and be ready to retract/replace.

Likely follow-ups
  • How do you guardrail a streamed response?
Red-flag answer pattern
Streaming safety-critical structured output with no post-validation.
QFew-shot vs zero-shot — how do you decide?BeginnerscreeningMidfreq: mediumBoth
Model answer

Start zero-shot. Add few-shot examples when you need consistent format/style or the task is ambiguous. Use a small number of high-quality, diverse examples.

Strong senior-level answer

I add examples only to fix a specific failure I've observed, and I measure the token-cost/quality trade. Often a clearer instruction or a schema beats adding examples.

Likely follow-ups
  • What's the downside of too many examples?
Red-flag answer pattern
Stuffing 20 examples 'to be safe' without measuring.
QHow do you design a system prompt vs a user prompt?Intermediatetechnical deep-diveSeniorfreq: mediumBoth
Model answer
  • System prompt — stable role, rules, output contract, safety constraints. Set once, rarely changes per request.
  • User prompt — the task and the per-request data (the question, the retrieved context).

Keep instructions in the system prompt and untrusted/variable content in the user turn.

Strong senior-level answer

A senior keeps untrusted content (retrieved docs, user text) clearly separated from authoritative instructions — that separation is also the first line of defense against prompt injection.

Likely follow-ups
  • Where do you put retrieved RAG context?
  • How does this relate to injection defense?
Red-flag answer pattern
Mixing instructions and untrusted data together so injected text carries authority.
QWalk me through calling an LLM API cleanly from Python (FastAPI).IntermediatecodingSeniorfreq: highBoth
Model answer
  • Wrap the provider client in a small service module (not scattered calls).
  • Use a Pydantic schema for structured output; validate the response.
  • Set timeouts and retries with backoff; handle rate limits.
  • Log the request/response (with token counts) for observability; never log secrets/PII raw.
  • Expose it as a typed FastAPI endpoint.
Strong senior-level answer

A senior treats the LLM as an unreliable network dependency — timeouts, retries, fallback model, and a clean seam so the provider is swappable — and makes prompts versioned config, not literals.

Likely follow-ups
  • How do you make this testable?
  • How do you add a fallback model?
Red-flag answer pattern
Calling the SDK inline everywhere with no validation, timeout, or error handling.

Common mistakes

  • Regexing JSON out of prose
  • Burying prompts as string literals with no versioning
  • Streaming output you then can't validate

Revision checklist

  • I can force + validate structured output
  • I know when to stream and its risks
  • I version my prompts

Embeddings & Vector Search

The retrieval substrate under RAG and agent memory: what an embedding is, how vector databases find neighbors fast (ANN/HNSW), and why production retrieval is almost always hybrid search plus a reranker — not pure cosine similarity.

Core for RAGHigh frequency
AudienceEngineers building retrieval into LLM apps.
GoalExplain embeddings, choose a vector store, and design hybrid retrieval with reranking.
You'll learn
  • What embeddings are and why cosine similarity works
  • ANN indexes (HNSW) and the recall/latency tradeoff
  • Choosing a vector DB (pgvector vs Qdrant vs FAISS)
  • Hybrid search (dense + BM25) and reranking
  • Why semantic-only retrieval fails on names, IDs and keywords

Why this topic shows up in interviews

Retrieval quality is the single biggest driver of RAG quality, so interviewers probe it hard. 'Why is your RAG returning irrelevant chunks?' is a near-guaranteed question, and the answer lives here.

Interview answers — 60-second & 3-minute

60 sec The crisp interview answer

“An embedding maps text to a vector so that semantically similar text lands nearby; we retrieve by nearest-neighbor (cosine) search, made fast with an ANN index like HNSW. In production I rarely use pure semantic search — I combine it with keyword/BM25 (hybrid) so exact terms, names and IDs aren't missed, then rerank the top candidates with a cross-encoder for precision.”

3 min The deeper answer that signals seniority

An embedding model maps text into a high-dimensional vector where distance encodes semantic similarity. To search millions of vectors we don't scan linearly — we use approximate nearest-neighbor indexes (HNSW is the common one), trading a little recall for big latency wins, tunable via parameters like ef_search.

The production reality: dense embeddings are great at meaning but weak at exact tokens — part numbers, names, acronyms. So I run hybrid search: dense + sparse (BM25), fuse the scores (e.g. reciprocal rank fusion), then rerank the top ~50 with a cross-encoder to get the top ~5 that actually go in context. Choice of store depends on scale and ops: pgvector if I already run Postgres and scale is moderate; Qdrant/Weaviate for larger, filter-heavy workloads; FAISS for in-process/offline.

Core concept

Embeddings & similarity

Text → vector. Cosine similarity (angle between vectors) measures semantic closeness. Normalize vectors and cosine ≈ dot product.

python
from sentence_transformers import SentenceTransformer
model = SentenceTransformer("BAAI/bge-small-en-v1.5")
vecs = model.encode(["reset my password", "how do I change my login"], normalize_embeddings=True)
import numpy as np
print(float(vecs[0] @ vecs[1]))  # high cosine -> semantically close

ANN indexes

Exact nearest-neighbor over millions of vectors is too slow. HNSW builds a navigable small-world graph; you trade exactness for speed. Key knobs: index build (M, ef_construction) and query (ef_search) — higher ef_search = better recall, more latency.

🎯
Hybrid + rerank is the default
If you only remember one thing: production retrieval = dense + sparse, fused, then reranked. Pure cosine-only retrieval is the #1 cause of 'my RAG returns junk'.

QYour RAG returns irrelevant chunks. Walk me through how you debug retrieval.Advancedtechnical deep-diveSeniorfreq: highBoth
Model answer

Inspect what's actually retrieved for failing queries. Check chunking (too big/small?), embedding model fit for the domain, whether exact terms are missed (add BM25/hybrid), and whether a reranker would fix ordering. Measure retrieval with recall@k / MRR on a labeled set.

Strong senior-level answer

I separate retrieval failures from generation failures first — log the retrieved chunks per query. Common root causes in order: no hybrid search (misses IDs/names), bad chunking that splits the answer, wrong/under-trained embedding model for the domain, and missing reranking. I fix with a labeled query set and recall@k as the target metric, so changes are measured, not guessed.

Likely follow-ups
  • How do you build the labeled retrieval set?
  • When would you fine-tune the embedding model?
  • Dense vs sparse for a parts catalog?
Red-flag answer pattern
Jumping to 'use a bigger LLM' for what is a retrieval problem.
Related hands-on task / project

Add hybrid search + a reranker to your RAG app; report recall@5 before/after.

QWhat is an ANN index and what do you trade for speed?Intermediatetechnical deep-diveSeniorfreq: highBoth
Model answer

Approximate nearest neighbor (e.g. HNSW) finds close-enough neighbors without scanning everything. You trade a little recall for large latency gains; ef_search tunes that trade.

Strong senior-level answer

I pick parameters from an SLO: set a latency budget, then raise ef_search until recall@k plateaus within budget. I also account for build time and memory (M, ef_construction) and re-index strategy as the corpus grows.

Likely follow-ups
  • How does HNSW differ from IVF?
  • What happens to recall as you scale to 100M vectors?
Red-flag answer pattern
Believing ANN is exact / lossless.

QDense vs sparse retrieval — when does each win?Intermediatetechnical deep-diveSeniorfreq: highBoth
Model answer

Dense (embeddings) wins on paraphrase and meaning; sparse (BM25) wins on exact tokens — IDs, names, codes, rare terms. Hybrid combines both.

Strong senior-level answer

I default to hybrid because real corpora mix both needs. For a legal/finance/parts domain with lots of exact identifiers, I weight sparse higher; for conversational FAQ, dense. I validate the weighting on a labeled set rather than guessing.

Likely follow-ups
  • How do you fuse the two score lists?
Red-flag answer pattern
Insisting embeddings alone are always sufficient.
QHow do you choose a vector database?Intermediatesystem designSeniorfreq: mediumBoth
Model answer

By scale, filtering needs, ops appetite and existing stack. pgvector if you already run Postgres at moderate scale; a dedicated store (Qdrant/Weaviate/Milvus) for large, filter-heavy workloads; FAISS for in-process/offline; managed if you don't want infra.

Strong senior-level answer

I start from constraints: data residency/compliance, metadata filtering, multi-tenancy, and whether the team can operate another stateful service. I avoid premature adoption of a heavy vector DB — pgvector covers a surprising amount and keeps one fewer system to run.

Likely follow-ups
  • Why not always use a managed service?
  • How do you handle multi-tenant isolation?
Red-flag answer pattern
Picking the trendiest DB with no reference to scale or ops.

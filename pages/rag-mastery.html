RAG Mastery

The single most-tested GenAI skill. This is the end-to-end pipeline — chunk → embed → retrieve → rerank → ground → generate with citations → evaluate — plus the failure modes and tradeoffs that separate a demo from a production RAG system.

Why this topic shows up in interviews

RAG is how most companies put their own data into an LLM without fine-tuning. It's the default architecture, so it's the default interview. Expect a full pipeline walkthrough plus deep follow-ups on chunking, retrieval quality and evaluation.

Interview answers — 60-second & 3-minute

60 sec The crisp interview answer

“RAG grounds an LLM in your data: chunk documents, embed them, retrieve the most relevant chunks for a query (hybrid search + reranking), put them in the prompt, and generate an answer with citations — instructing the model to say 'I don't know' if the context doesn't support an answer. It beats fine-tuning when knowledge changes often or must be sourced/auditable. The hard parts are chunking and retrieval quality, and you must evaluate both retrieval and generation.”

3 min The deeper answer that signals seniority

RAG has two halves people conflate. Indexing (offline): load → chunk → embed → store in a vector DB with metadata. Serving (online): embed the query → retrieve (ideally hybrid dense+sparse) → rerank → assemble a grounded prompt → generate with citations → validate.

The quality of a RAG system is dominated by retrieval, and retrieval is dominated by chunking and hybrid+rerank. So I instrument both stages: retrieval metrics (recall@k, MRR, nDCG on a labeled set) and generation metrics (faithfulness/groundedness, answer relevance). For trust I require citations and an explicit 'insufficient context → say so' path.

On RAG vs fine-tuning: RAG for knowledge that changes, needs attribution, or is large; fine-tuning for behavior/format/style. They compose — you can fine-tune tone and RAG the facts. I almost always reach for RAG first because it's cheaper to update and auditable.

Chunking is the highest-leverage RAG decision

QWalk me through a production RAG pipeline end to end.Intermediatetechnical deep-diveSeniorfreq: highBoth
Model answer

Offline: load, chunk (with overlap/structure), embed, store with metadata. Online: embed query, hybrid retrieve, rerank, assemble a grounded prompt with citations and an abstain instruction, generate, validate. Evaluate retrieval (recall@k) and generation (faithfulness) separately.

Strong senior-level answer

I emphasize the production seams: metadata for filtering and access control, a reranker to keep context small and precise, citations for auditability, an abstain path when retrieval is weak, caching of embeddings and frequent queries, and observability (trace each query's retrieved chunks). And I'd state my eval plan up front because that's what makes it maintainable.

Likely follow-ups
  • Where does it most commonly fail?
  • How do you handle access control / multi-tenant data?
  • How do you keep the index fresh?
Red-flag answer pattern
Describing only 'embed → cosine search → stuff into prompt' with no reranking, citations, or evaluation.
Related hands-on task / project

Build the full pipeline in your RAG app including citations and an eval harness.

QHow do you choose a chunking strategy?Advancedtechnical deep-diveSeniorfreq: highBoth
Model answer

Start with fixed-size + overlap as a baseline; move to structure-aware for well-structured docs; use parent-document/sentence-window when you need precise matching but enough surrounding context to answer. Validate with retrieval metrics.

Strong senior-level answer

I treat chunking as an experiment driven by a labeled query set, not a default. I check whether answers span chunk boundaries (a sign chunks are too small or overlap too low) and whether embeddings are diluted (chunks too big). Parent-document retrieval is my common landing spot: embed sentences/small chunks for precision, return the parent section for context.

Likely follow-ups
  • What overlap do you use and why?
  • How does chunk size interact with the embedding model's max length?
Red-flag answer pattern
A fixed '512 tokens always' answer with no reference to document structure or measurement.
QRAG vs fine-tuning — when do you pick which?Intermediatetechnical deep-diveSeniorfreq: highBoth
Model answer

RAG for knowledge that changes, is large, or must be sourced/audited. Fine-tuning for consistent behavior, format, tone, or domain phrasing. They compose.

Strong senior-level answer

Default to RAG because updates are a re-index, not a retraining run, and answers are auditable. I'd only fine-tune when prompting+RAG can't hit a behavioral/format/latency target, and even then prefer lightweight (LoRA) fine-tuning. I make the call on cost-to-update and need-for-attribution, not hype.

Likely follow-ups
  • Can you do both?
  • What about fine-tuning the embedding model instead?
Red-flag answer pattern
'Fine-tune the LLM on our docs' as the default for a knowledge problem.
QHow do you evaluate a RAG system?Advancedtechnical deep-diveSeniorfreq: highBoth
Model answer

Separately. Retrieval: recall@k, MRR, nDCG on a labeled query→relevant-doc set. Generation: faithfulness/groundedness (claims supported by context), answer relevance, and correctness vs a golden set. Combine with LLM-as-judge for scale.

Strong senior-level answer

I build a small golden dataset early and treat eval as CI: every retrieval/prompt change runs it. I split metrics so I know whether a regression is retrieval or generation. For groundedness I use an LLM-as-judge with a calibrated rubric and spot-check it against human labels. In production I sample live traffic for the same checks.

Likely follow-ups
  • How big should the golden set be?
  • How do you trust LLM-as-judge?
  • RAGAS-style metrics?
Red-flag answer pattern
'We eyeballed a few answers and it looked good.'
Related hands-on task / project

Add recall@k + a faithfulness judge to your RAG app and wire it into CI.

QRetrieval returns good chunks but the answer is still wrong. What's happening?Advancedtechnical deep-diveSeniorfreq: mediumBoth
Model answer

It's a generation problem: the prompt may not constrain the model to the context, context ordering/overflow ('lost in the middle'), conflicting chunks, or the model ignoring the abstain instruction. Tighten grounding instructions, reduce/rerank context, and check for context overflow.

Strong senior-level answer

I confirm the right chunks are present, then look at: prompt not forcing context-only answers, too many chunks causing dilution, contradictory sources with no resolution strategy, and missing citations so I can't tell what it used. Fixes: fewer reranked chunks, explicit cite-or-abstain, and a conflict-resolution instruction. I verify with a faithfulness metric, not by re-reading one answer.

Likely follow-ups
  • What's 'lost in the middle' and how do you mitigate it?
Red-flag answer pattern
Blaming the model and swapping to a bigger one without isolating retrieval vs generation.
Source / evidence

Liu et al., 2023 — Lost in the Middle (arXiv)

QHow do you choose chunk size?Advancedtechnical deep-diveSeniorfreq: highBoth
Model answer

There's no universal number — you tune it against a labeled query set. Rules of thumb:

  • Smaller chunks (≈200–400 tokens) → precise matching, but answers may get split across chunks.
  • Larger chunks (≈800–1000 tokens) → more context per chunk, but diluted embeddings and wasted context window.
  • Respect the embedding model's max input length.
  • Start ~400–512 tokens with overlap, then measure recall@k and adjust.
Strong senior-level answer

A senior decouples the retrieval unit from the context unit — embed small chunks for precise matching but return the larger parent passage for enough context to answer (parent-document retrieval) — so they don't have to compromise.

Likely follow-ups
  • How does chunk size interact with overlap?
  • Why not just use huge chunks now that windows are large?
Red-flag answer pattern
A fixed number with no measurement and no awareness of the size/precision trade.
Related hands-on task / project

Sweep 3 chunk sizes on your golden set and compare recall@5.

QHow do you handle overlapping chunks?Intermediatetechnical deep-diveSeniorfreq: highBoth
Model answer

Overlap means consecutive chunks share some text (typically ~10–20%). It stops an answer that sits on a chunk boundary from being cut in half.

  • Too little overlap → boundary answers get split and missed.
  • Too much overlap → duplicate content inflates the index, cost, and can return near-duplicate chunks.
Strong senior-level answer

A senior notes overlap is a band-aid for fixed-size splitting and prefers structure-aware or parent-document chunking where overlap matters less, and de-duplicates near-identical retrieved chunks before they hit the context.

Likely follow-ups
  • What overlap % do you use and why?
  • How do you avoid returning duplicate chunks?
Red-flag answer pattern
Adding large overlap everywhere 'to be safe' without considering index bloat.
QWhy use reranking, and why a cross-encoder?Advancedtechnical deep-diveSeniorfreq: highBoth
Model answer

First-pass retrieval (vector/keyword) is fast but imprecise, so it returns many roughly-relevant chunks. A reranker reorders the top candidates by true relevance and keeps the best few for the prompt.

A cross-encoder reads the query and a chunk together, so it judges relevance far more accurately than comparing separate embeddings — at the cost of speed, which is why you only rerank the top ~50, not the whole corpus.

Strong senior-level answer

A senior uses reranking to keep the context small and high-precision (cheaper, less 'lost in the middle') and measures the recall@k / faithfulness lift to justify the extra latency.

Likely follow-ups
  • Why not cross-encode the whole corpus?
  • How much latency does reranking add?
Red-flag answer pattern
Putting 20 un-reranked chunks into the prompt and hoping the model sorts it out.
QHow do you add citations, and why do they matter?Intermediatetechnical deep-diveSeniorfreq: highBoth
Model answer

Pass each retrieved chunk with an id and source metadata (file, page, URL), instruct the model to cite the id it used for each claim (e.g. [1]), and map those ids back to sources in the response.

Citations matter because they make answers auditable, let users verify, and let you detect hallucination (a claim with no citation is suspect).

Strong senior-level answer

A senior validates that cited chunks actually support the claim (not just that a citation exists) and uses citation presence as a guardrail — block or flag answers that make uncited factual claims.

Likely follow-ups
  • How do you verify the citation actually supports the claim?
  • How do you cite a specific page or clause?
Red-flag answer pattern
Treating citations as decoration without checking they back the claim.
Related hands-on task / project

Add id-tagged context + citation mapping to your RAG app.

QWhat can go wrong in production RAG?Advancedsystem designSeniorfreq: highBoth
Model answer

The common production failures:

  • Retrieval misses (semantic-only, bad chunking) → wrong/empty context.
  • Hallucination when context is weak and there's no abstain path.
  • Stale index — documents changed but weren't re-indexed.
  • Cross-permission data leaks — retrieving docs a user shouldn't see.
  • Indirect prompt injection via a poisoned document.
  • Latency/cost blowups from huge contexts.
Strong senior-level answer

A senior maps each to a mitigation (hybrid+rerank, abstain gate, freshness pipeline, access control at retrieval, untrusted-content handling, caching) and instruments them so failures are visible in traces, not discovered by users.

Likely follow-ups
  • How do you detect a stale index?
  • How do you stop cross-tenant leaks?
Red-flag answer pattern
Only describing the happy path with no failure modes.

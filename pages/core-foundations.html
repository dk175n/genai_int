Why this topic shows up in interviews

Screening rounds almost always open here. Getting tokens, context windows and decoding right signals you understand the substrate; getting hallucination right sets up every RAG/eval/guardrails answer later.

Interview answers — 60-second & 3-minute

60 sec The crisp interview answer

“An LLM is a next-token predictor: given a sequence of tokens it outputs a probability distribution over the next token, sampled repeatedly. It's stateless — the only memory is what's in the context window this call. Temperature controls randomness of sampling. Hallucination is intrinsic because the model optimizes for plausible continuations, not truth — which is exactly why we ground it with retrieval.”

3 min The deeper answer that signals seniority

Start with the mechanism: a transformer takes tokenized text, and through self-attention produces a probability distribution over the vocabulary for the next token. We sample one, append it, repeat. That's it — there's no database lookup, no reasoning engine, just learned statistical structure.

Three consequences interviewers care about: (1) statelessness — each API call only knows the tokens you send, which is why chat history and RAG context must be re-sent every time; (2) the context window bounds how much it can attend to, and tokens cost money and latency; (3) hallucination is structural — when the most plausible continuation isn't the true one, the model still produces it confidently. Grounding (RAG), constrained decoding, and evaluation are how we manage that.

Core concept

Next-token prediction

The model is a function from a token sequence to a distribution over the next token. Decoding repeats this. Everything else — chat, RAG, agents — is scaffolding around this single primitive.

python
# A chat call is just: tokens in -> next tokens out, one shot.
from anthropic import Anthropic
client = Anthropic()
resp = client.messages.create(
    model="claude-opus-4-8",
    max_tokens=512,
    messages=[{"role": "user", "content": "Summarize this contract: ..."}],
)
print(resp.content[0].text)
# The model retains NOTHING after this call. State = what you send next time.

Tokens & the context window

Text is split into tokens (~4 chars / 0.75 words in English). The context window is the max tokens (input + output) per call. Bigger context = more cost and latency, and quality can degrade in the middle of very long contexts (the 'lost in the middle' effect).

Hallucination is intrinsic
The model maximizes plausibility, not truth. You cannot prompt it away entirely. You manage it: ground answers in retrieved sources, require citations, validate outputs, and evaluate groundedness. This is the thread that connects RAG, guardrails and eval.

QWhy does an LLM hallucinate, and how do you reduce it in production?Intermediatetechnical deep-diveSeniorfreq: highBoth
Model answer

It generates the most probable continuation, which isn't always true. Reduce it by grounding answers in retrieved context (RAG), requiring citations and 'I don't know' when context is insufficient, lowering temperature for factual tasks, and validating outputs.

Strong senior-level answer

I treat it as a measurable property, not a vibe. I add a groundedness/faithfulness metric (does every claim trace to a retrieved chunk?), gate responses on a confidence/coverage signal, and monitor hallucination rate in production via sampled LLM-as-judge checks. Prompting helps at the margin; architecture and evaluation are what actually move the number.

Likely follow-ups
  • How do you measure groundedness?
  • What if retrieval returns nothing relevant?
  • Does a bigger model hallucinate less?
Red-flag answer pattern
“You just tell it not to hallucinate in the prompt.” Signals no production experience.
Related hands-on task / project

Add a 'cite or say I don't know' instruction to your RAG app and measure unsupported-claim rate before/after.

QWhat is the context window and why does it matter for cost and quality?BeginnerscreeningMidfreq: highBoth
Model answer

It's the max tokens (input + output) the model can process per call. It matters because everything — system prompt, history, retrieved context — competes for it; you pay per token and latency grows with it.

Strong senior-level answer

I budget the window deliberately: cap history, compress or rerank retrieved context to the top-k that fits, and watch for 'lost in the middle' degradation in long contexts. Bigger windows aren't free quality — placement and relevance of context matter more than raw size.

Likely follow-ups
  • How do you decide how many chunks to put in context?
  • What's 'lost in the middle'?
Red-flag answer pattern
Thinking the model 'remembers' previous calls without you resending context.
QWhen would you change temperature, and to what?BeginnerscreeningMidfreq: mediumBoth
Model answer

Low (0–0.3) for extraction, classification, JSON, and anything you'll evaluate deterministically; higher (0.7–1.0) for ideation/creative tasks where diversity helps.

Strong senior-level answer

For production features I default to low temperature for reproducibility and easier evaluation, and pair it with structured output. I only raise it where diversity is the product (e.g. generating multiple draft options) and then sample N and rank.

Likely follow-ups
  • How does temperature interact with evaluation reproducibility?
Red-flag answer pattern
Using high temperature for structured/extraction tasks.
QWhat is a token, and why should an engineer care?BeginnerscreeningMidfreq: highBoth
Model answer

A token is a chunk of text the model processes (~4 characters / ~0.75 words in English). Models read and write in tokens, not words.

Why it matters:

  • You pay per token (input + output) — cost is token-driven.
  • The context window is measured in tokens.
  • Latency grows with tokens generated.
Strong senior-level answer

A senior estimates token cost before shipping (counts tokens of a representative prompt + context), and trims context as a primary cost lever — not as an afterthought.

Likely follow-ups
  • How many tokens is your typical RAG prompt?
  • How does tokenization differ across languages?
Red-flag answer pattern
Equating tokens exactly with words, or not knowing you pay per token.
QDoes a bigger model always give better answers?Intermediatetechnical deep-diveSeniorfreq: mediumBoth
Model answer

No. A bigger model is more capable but slower and more expensive. For many tasks (classification, extraction, simple RAG answering) a smaller model is good enough and far cheaper/faster.

Choose by measuring quality on your eval set against cost and latency budgets.

Strong senior-level answer

A senior routes by task difficulty — small model for easy traffic, large for hard cases — and proves the smaller model holds quality on the eval set before downgrading. Bigger ≠ better, cheaper-that-passes-eval wins.

Likely follow-ups
  • How do you decide the routing threshold?
  • When is fine-tuning a small model better than a big model?
Red-flag answer pattern
'Always use the biggest model to be safe' with no cost/latency consideration.

Hands-on task

Write a script that calls an LLM with temperature 0 and 1 on the same factual prompt 5×; observe variance. Then count tokens of a long document with a tokenizer and estimate cost.

Portfolio connection

This is the 'why' behind your RAG app: you ground because the model hallucinates. Say that explicitly in your README.

Common mistakes

  • Saying the model 'remembers' across calls
  • Treating hallucination as a prompt bug
  • Using high temperature for extraction

Revision checklist

  • I can explain why grounding reduces hallucination
  • I know when to change temperature

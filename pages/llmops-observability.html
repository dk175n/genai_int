LLMOps & Observability

Running GenAI in production: tracing every request, controlling latency and cost (caching, streaming, model routing, fallbacks), and monitoring quality and drift. This is where backend engineers have a real edge — and where applied/platform interviews dig in.

Backend edgePlatform loops
AudienceEngineers responsible for GenAI features in production.
GoalInstrument, optimize and operate an LLM system and defend the tradeoffs.
You'll learn
  • Tracing LLM apps (spans, Langfuse/LangSmith concepts)
  • Latency optimization: streaming, parallelism, smaller models
  • Cost optimization: caching, routing, prompt compression
  • Fallback models & graceful degradation
  • Monitoring: quality, cost, latency, drift
  • Deployment patterns for LLM services

Why this topic shows up in interviews

LLM calls are slow, expensive and non-deterministic — the operational concerns are real and constant. Platform/applied interviews probe whether you can keep a GenAI feature fast, cheap, observable and reliable, not just functional.

Interview answers — 60-second & 3-minute

60 sec The crisp interview answer

“LLMOps is treating LLM features like production systems. I trace every request end-to-end (retrieval, prompts, tool calls, tokens, cost, latency) with something like Langfuse or LangSmith. I cut cost with caching, model routing (small model for easy cases, large for hard), and prompt trimming; I cut latency with streaming and parallel retrieval. I add fallback models for provider failures, and I monitor quality, cost, latency and drift with alerts — the same rigor as any service, plus token economics.”

3 min The deeper answer that signals seniority
Observability: a single LLM 'request' is really a tree — retrieval, rerank, one or more model calls, tool calls. I trace it as spans capturing inputs, outputs, token counts, cost and latency per step, so I can debug 'why was this answer bad/slow/expensive?'. That's what Langfuse/LangSmith give you.Cost & latency levers: caching (exact and semantic) for repeated queries; model routing so cheap models handle easy traffic; prompt/context compression and fewer reranked chunks; streaming for perceived latency; running retrieval and independent tool calls in parallel; batching offline work.Reliability: fallback models and providers, timeouts and retries with backoff, rate-limit handling, and graceful degradation (e.g. serve a cached or smaller-model answer under load). Monitoring: dashboards + alerts on cost per request, p95 latency, error/abstain rates, and sampled quality scores, plus drift detection on inputs and outputs.

# Minimal trace + cache + fallback sketch
@trace("answer")
def answer(q):
    if (hit := cache.get(q)): return hit            # exact cache
    ctx = retrieve(q)                                # traced span
    try:
        out = call(PRIMARY_MODEL, prompt(q, ctx), timeout=8)
    except (Timeout, ProviderError):
        out = call(FALLBACK_MODEL, prompt(q, ctx))   # graceful degradation
    cache.set(q, out, ttl=3600)
    return out

QHow do you reduce the cost of an LLM feature without hurting quality?Advancedsystem designSeniorfreq: highBoth
Model answer

Cache repeated queries (exact + semantic), route easy traffic to cheaper models and reserve the large model for hard cases, trim prompts/context, and batch offline work. Validate any model downgrade against the eval set so quality doesn't silently drop.

Strong senior-level answer

I instrument cost per request first so I optimize where the money actually is. The biggest wins are usually caching and routing. Crucially, every cost optimization runs through the eval gate — I never trade quality blindly. I also separate one-time indexing cost from per-query cost in the analysis.

Likely follow-ups
  • How does semantic caching work and when is it risky?
  • How do you decide the routing threshold?
Red-flag answer pattern
Downgrading the model with no eval to confirm quality held.
Related hands-on task / project

Add caching + routing to your app and report cost/request before/after on the eval set.

QWhat do you trace in an LLM app and why?Intermediatetechnical deep-diveSeniorfreq: highBoth
Model answer

The full request tree: retrieved chunks, prompts, model calls, tool calls, token counts, cost and latency per step, plus quality scores. So you can debug bad/slow/expensive answers and monitor trends. Tools like Langfuse/LangSmith do this.

Strong senior-level answer

I attach a trace/session id end-to-end and log the retrieved context with each answer — that single thing makes most RAG debugging tractable, because I can see whether a failure was retrieval or generation. I also sample traces for offline judging so observability and evaluation share one pipeline.

Likely follow-ups
  • How do you debug a single bad answer from a trace?
  • How do you handle PII in traces?
Red-flag answer pattern
Logging only the final answer with no intermediate steps.
QHow do you keep an LLM feature reliable when the provider is slow or down?Advancedsystem designSeniorfreq: mediumBoth
Model answer

Timeouts with retries/backoff, a fallback model or provider, rate-limit handling, and graceful degradation (cached or smaller-model answer, or an honest 'try again'). Circuit breakers to avoid hammering a failing provider.

Strong senior-level answer

I design for provider failure as a normal event: health checks, a secondary provider with a known-acceptable quality floor (validated on eval), and degradation tiers. I also separate 'must answer now' paths from 'can queue' paths so I can shed load gracefully under incidents.

Likely follow-ups
  • How do you ensure the fallback model's quality is acceptable?
Red-flag answer pattern
Assuming one provider with no fallback or timeout.
QWhat metrics do you track for a GenAI app in production?Advancedsystem designSeniorfreq: highBoth
Model answer

Four families:

  • Quality — sampled faithfulness/relevance scores, abstain rate, user feedback (thumbs).
  • Cost — tokens and $ per request, per feature, per tenant.
  • Latency — p50/p95/p99, broken down per step (retrieval, model, tools).
  • Reliability — error rate, timeout/fallback rate, rate-limit hits.

Plus input/output drift over time.

Strong senior-level answer

A senior ties quality monitoring to the same judges that gate deploys, so production is continuously scored — not just availability. They alert on regressions and anomalies, not just thresholds.

Likely follow-ups
  • How do you measure quality without ground-truth labels in prod?
  • How do you detect drift?
Red-flag answer pattern
Monitoring only uptime and latency while ignoring answer quality.

QHow do you handle fallback models?Advancedsystem designSeniorfreq: mediumBoth
Model answer

Have a secondary model/provider ready and trigger it on timeout, error, or rate-limit from the primary.

  • Validate the fallback's quality on your eval set first — know its floor.
  • Use a circuit breaker so you stop hammering a failing provider.
  • Optionally degrade further: cached answer, smaller model, or an honest retry message.
Strong senior-level answer

A senior designs degradation tiers and treats provider failure as a normal event, with the fallback's acceptable-quality threshold known in advance, not assumed.

Likely follow-ups
  • How do you know the fallback is good enough?
  • Same prompt across providers — any gotchas?
Red-flag answer pattern
A fallback model that's never been evaluated, so quality silently tanks during incidents.
QHow do you monitor for hallucinations in production?Advancedtechnical deep-diveSeniorfreq: highBoth
Model answer

You can't label everything, so sample: run a faithfulness judge on a percentage of live traffic, comparing the answer to the retrieved context. Track the unsupported-claim rate over time and alert on spikes.

Combine with cheap signals: citation coverage, abstain rate, and user thumbs-down.

Strong senior-level answer

A senior wires this into the same trace pipeline (so each flagged answer is debuggable end-to-end) and feeds confirmed hallucinations back into the golden set.

Likely follow-ups
  • How do you trust the judge?
  • What sampling rate?
Red-flag answer pattern
Assuming hallucinations are caught by users reporting them.

QWalk me through debugging a single bad answer in production.Advancedtechnical deep-diveSeniorfreq: highBoth
Model answer

Pull the trace for that request and check, in order:

  • Retrieval — were the right chunks retrieved? (If not, it's a retrieval bug — chunking/hybrid/rerank.)
  • Prompt — what context and instructions were actually sent?
  • Generation — did the model ignore the context or hit a length/overflow issue?
  • Guardrails — did anything get filtered or altered?
Strong senior-level answer

A senior relies on having logged the retrieved context with the answer, so they isolate retrieval-vs-generation in seconds, then reproduce it as a golden-set case so it can't regress.

Likely follow-ups
  • What do you log to make this fast?
  • How do you turn it into a regression test?
Red-flag answer pattern
Re-running it a few times and guessing, with no trace to inspect.

Common mistakes

  • No tracing → unexplainable failures
  • Optimizing cost without an eval gate
  • Single provider, no fallback/timeout
  • Caching with no invalidation strategy

Revision checklist

  • I can describe a trace tree and why each span matters
  • I have a cost and a latency playbook
  • I can design fallback + graceful degradation
  • I monitor quality, cost, latency, drift

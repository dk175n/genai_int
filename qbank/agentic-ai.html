QHow does an agent / tool calling actually work?Intermediatetechnical deep-diveSeniorfreq: highBoth
Model answer

An agent is an LLM in a loop with tools. The cycle:

  • You give the model a set of tools (functions) with their input schemas.
  • It outputs a structured request to call one tool with some arguments.
  • Your code runs the real function and returns the result.
  • The result goes back into the conversation; the model decides the next step or gives a final answer.
  • Repeat until done — with a maximum step limit.

Key point: the model only proposes a call; your runtime actually executes it.

Strong senior-level answer

A senior frames it as 'model proposes, runtime disposes': the runtime is the trust boundary, so it validates the arguments, keeps tools least-privilege, and bounds the loop. That framing is what makes agents safe.

Likely follow-ups
  • What stops it from looping forever?
  • What if a tool fails?
Red-flag answer pattern
Thinking the LLM itself runs the code/tool.

QReAct vs planner-executor vs stateful graph?Advancedtechnical deep-diveSeniorfreq: mediumBoth
Model answer
  • ReAct — reason and act one step at a time, adapting as it goes. Flexible but can wander and is hard to predict.
  • Planner-executor — make a full plan first, then execute the steps. Predictable and easier to debug.
  • Stateful graph (LangGraph) — model the agent as nodes/edges with explicit state, checkpoints and human-in-the-loop. The production-grade option.
Strong senior-level answer

A senior leans toward planner-executor or an explicit graph in production because predictability, cost control and observability matter more than raw flexibility.

Likely follow-ups
  • How do you bound cost in ReAct?
Red-flag answer pattern
Treating ReAct as always the best choice.
QWhy do agents fail in production, and how do you make them reliable?Advancedtechnical deep-diveSeniorfreq: highBoth
Model answer

Common failures:

  • Looping forever, or calling the wrong tool / wrong arguments.
  • One bad early step cascading into a wrong result.
  • Cost blowups from too many calls.
  • Prompt injection hidden in tool outputs.

Fixes:

  • Bound the loop (max steps) and set a token/cost budget.
  • Validate every tool call against its schema.
  • Make tools least-privilege and idempotent; add timeouts/retries.
  • Guard inputs and outputs; treat tool results as untrusted.
Strong senior-level answer

A senior adds explicit state with checkpointing (retry a step, not the whole run), human-in-the-loop on risky actions, and full tracing so the decision path is debuggable.

Likely follow-ups
  • Where do you put human-in-the-loop?
Red-flag answer pattern
No mention of bounding the loop or budgets.
QWhen would you NOT build an agent?Intermediatesystem designSeniorfreq: mediumBoth
Model answer

When the steps are known in advance. Then a plain deterministic workflow (with maybe one LLM call per step) is cheaper, faster, more testable, and more reliable than a free-running agent loop.

Strong senior-level answer

A senior defaults to the least-agentic solution that works and only escalates to a full agent when the path is genuinely dynamic at runtime. That restraint signals maturity.

Likely follow-ups
  • Give an example you'd build as a workflow, not an agent.
Red-flag answer pattern
Reaching for an agent for everything because it's impressive.

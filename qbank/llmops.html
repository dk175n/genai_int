QHow do you reduce the cost of an LLM feature without hurting quality?Advancedsystem designSeniorfreq: highBoth
Model answer
  • Cache repeated queries (exact matches and semantically similar ones).
  • Route easy requests to a cheaper/smaller model, reserve the big model for hard ones.
  • Trim the prompt/context to what's needed.
  • Batch offline work.
  • Always re-run your evaluation after a change so quality doesn't silently drop.
Strong senior-level answer

A senior measures cost-per-request first to optimize where the money actually is, and runs every cost change through the eval gate — never trading quality blindly.

Likely follow-ups
  • How does semantic caching work and when is it risky?
Red-flag answer pattern
Downgrading the model with no eval to confirm quality held.
Related hands-on task / project

Monitoring Dashboard.

QWhat do you trace in an LLM app, and why?Intermediatetechnical deep-diveSeniorfreq: highBoth
Model answer

Trace the whole request as a tree of steps:

  • Which chunks were retrieved.
  • The prompts and model calls.
  • Any tool calls.
  • Token counts, cost and latency per step.
  • Quality scores.

This lets you debug why an answer was bad, slow, or expensive. Tools like Langfuse/LangSmith do this.

Strong senior-level answer

A senior attaches one trace id end-to-end and logs the retrieved context with every answer — that single habit makes most RAG debugging a quick lookup (was it retrieval or generation that failed?).

Likely follow-ups
  • How do you handle PII in traces?
Red-flag answer pattern
Logging only the final answer with no intermediate steps.

QHow do you keep the feature reliable when the model provider is slow or down?Advancedsystem designSeniorfreq: mediumBoth
Model answer
  • Set timeouts and retries with backoff.
  • Have a fallback model or a second provider.
  • Handle rate limits gracefully.
  • Degrade gracefully — serve a cached or smaller-model answer, or an honest 'try again', instead of failing hard.
  • Use circuit breakers so you stop hammering a failing provider.
Strong senior-level answer

A senior designs provider failure as a normal event, with a fallback whose quality they've already validated on the eval set, and tiers of degradation under load.

Likely follow-ups
  • How do you ensure the fallback model's quality is acceptable?
Red-flag answer pattern
Assuming one provider with no fallback or timeout.
QHow do you cut latency for a RAG or agent app?Intermediatetechnical deep-diveSeniorfreq: mediumBoth
Model answer
  • Stream the answer so the user sees text immediately.
  • Run retrieval and independent tool calls in parallel.
  • Use a smaller/faster model where quality allows (check on eval).
  • Tighten retrieval so you process fewer chunks.
  • Cache frequent results.
Strong senior-level answer

A senior optimizes p95 latency against a target SLO and measures per-step latency from traces first, so they fix the actual bottleneck.

Likely follow-ups
  • What does streaming make harder?
Red-flag answer pattern
Guessing at fixes without measuring where the time goes.

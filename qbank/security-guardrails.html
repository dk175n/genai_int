QWhat is prompt injection and how do you defend against it?Advancedtechnical deep-diveSeniorfreq: highBoth
Model answer

Prompt injection is when malicious instructions trick the model into ignoring its real instructions. Two forms:

  • Direct — the user types 'ignore previous instructions and…'.
  • Indirect — hidden instructions inside a retrieved document or tool output that the model then follows. This is the dangerous one for RAG/agents.

Defenses (layered, no single fix):

  • Treat all retrieved/tool content as untrusted data, never as instructions.
  • Validate outputs before using them.
  • Keep tool permissions least-privilege so a hijacked model can't do damage.
Strong senior-level answer

A senior emphasizes indirect injection, assumes some attempts will succeed, and limits the blast radius — plus adds injection cases to the eval set as a permanent gate.

Likely follow-ups
  • How would you exfiltrate data from a naive RAG agent?
Red-flag answer pattern
'Just tell the model in the system prompt to ignore injection attempts.'

QHow do you stop a RAG system from leaking data a user shouldn't see?Advancedsystem designSeniorfreq: highBoth
Model answer

Enforce access control at the retrieval step, not in the prompt:

  • Tag each chunk with permission metadata (which group/role can see it).
  • Filter the search by the requesting user's permissions, so restricted documents are never even candidates.
  • For hard boundaries, use separate indexes per tenant.
  • Add an output PII/secret check as a backstop.
Strong senior-level answer

A senior carries the source system's permissions into the chunk metadata at ingestion and tests 'restricted query returns nothing' as a hard invariant. The model is the last thing they'd trust to enforce permissions.

Likely follow-ups
  • Per-document permissions vs separate indexes — when each?
Red-flag answer pattern
Relying on a prompt instruction to keep restricted data hidden.
QHow do you secure an agent that can take actions (send email, run SQL)?Advancedsystem designSeniorfreq: mediumBoth
Model answer
  • Give each tool the minimum capability it needs (e.g. read-only SQL with row limits).
  • Validate every tool argument.
  • Require confirmation or human approval for destructive/irreversible actions.
  • Control what the agent can reach (egress) and log every action for audit.
  • Treat tool outputs as untrusted (injection risk).
Strong senior-level answer

A senior makes safety come from what the tools can't do, not from the model behaving — because the model can be manipulated.

Likely follow-ups
  • How do you make a SQL tool safe?
Red-flag answer pattern
Giving the agent broad write/delete access and trusting it to behave.
QDirect vs indirect injection — which is worse for agents?Advancedtechnical deep-diveSeniorfreq: emergingBoth
Model answer

Indirect is worse. A poisoned document or web page can contain hidden instructions that the agent reads and then acts on — sending data or taking actions — without the user being malicious at all.

Strong senior-level answer

A senior designs so that even if the model is hijacked, the tools and permissions prevent real damage, and tests for it explicitly.

Likely follow-ups
  • How would you test for indirect injection?
Red-flag answer pattern
Only knowing about 'ignore previous instructions'.

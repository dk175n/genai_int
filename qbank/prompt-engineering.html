QHow do you get reliable structured (JSON) output from an LLM?IntermediatecodingSeniorfreq: highBoth
Model answer

Don't try to extract JSON from free text with string parsing — it breaks. Instead:

  • Define a schema (e.g. a Pydantic model).
  • Use the provider's structured-output / tool-calling feature and force it to use that schema.
  • Validate the result against the schema.
  • If validation fails, retry once or twice, feeding the error back to the model.
Strong senior-level answer

A senior treats the schema as the single source of truth, bounds the retries, keeps temperature low, and logs the schema-failure rate as a quality signal (a rising rate usually means a prompt or schema drift).

Likely follow-ups
  • What do you do if it never produces valid output?
  • How does this interact with streaming?
Red-flag answer pattern
Parsing JSON out of prose with regex/string operations and no validation.

QWhen do you add few-shot examples?BeginnerscreeningMidfreq: mediumBoth
Model answer

Start with no examples (zero-shot). Add a few examples when:

  • You need a consistent output format or style.
  • The task is ambiguous and examples clarify it.
  • Use a small number (1–3) of high-quality, varied examples — not a pile of mediocre ones.
Strong senior-level answer

A senior adds examples to fix a specific failure they've actually observed, and measures the trade-off (more examples = more tokens = more cost/latency). Often a clearer instruction or a schema beats adding examples.

Likely follow-ups
  • What's the downside of too many examples?
Red-flag answer pattern
Stuffing 20 examples 'to be safe' without measuring whether they help.
QWhat is chain-of-thought and when is it worth it?Intermediatetechnical deep-diveSeniorfreq: mediumBoth
Model answer

Chain-of-thought = letting the model 'think step by step' before giving the final answer. It helps on multi-step reasoning (math, logic, planning).

  • Use it for genuinely reasoning-heavy tasks.
  • Don't use it for simple lookups — it just adds cost and latency.
  • Hide the reasoning from end users; keep it internal.
Strong senior-level answer

A senior notes the cost/latency trade-off and that modern reasoning models often handle this internally, so explicit CoT is needed less than it used to be.

Likely follow-ups
  • Do you show users the reasoning?
Red-flag answer pattern
Adding 'think step by step' to every prompt, including trivial ones.
QHow do you manage prompts in production?Advancedsystem designSeniorfreq: mediumBoth
Model answer

Treat prompts as configuration, not code comments:

  • Store them outside the code and version them.
  • A/B test changes instead of editing blindly.
  • Run them through your evaluation set before shipping — a prompt edit is a production change.
  • Be able to roll back to a previous prompt.
Strong senior-level answer

A senior keeps a prompt registry tied into the evaluation CI, so no prompt change ships without passing the golden set, and every version is traceable.

Likely follow-ups
  • How would you A/B test a prompt?
Red-flag answer pattern
Prompts buried as string literals in code with no versioning or testing.

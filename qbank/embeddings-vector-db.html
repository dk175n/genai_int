QWhat is an embedding and why does cosine similarity work?BeginnerscreeningMidfreq: highBoth
Model answer

An embedding turns a piece of text into a vector (a list of numbers) so that texts with similar meaning end up close together in that space.

Cosine similarity measures the angle between two vectors — a small angle means similar meaning. So to find relevant text, you find the vectors closest to the query's vector.

Strong senior-level answer

A senior notes you normally normalize the vectors, that the embedding model must fit your domain, and that embeddings capture meaning but miss exact tokens (IDs, names) — which is why keyword search is still needed alongside them.

Likely follow-ups
  • When do embeddings fail to find the right text?
Red-flag answer pattern
Can't explain what 'similar' means or how search uses it.

QDense vs sparse vs hybrid retrieval — what's the difference?Intermediatetechnical deep-diveSeniorfreq: highBoth
Model answer
  • Dense (embeddings) — great at meaning and paraphrase ('reset password' ≈ 'change my login').
  • Sparse (BM25 / keyword) — great at exact terms: IDs, names, codes, rare words.
  • Hybrid — run both, combine the results, then rerank. This is the production default because real questions need both.
Strong senior-level answer

A senior weights the two by domain (more keyword weight for legal/finance/part-numbers, more semantic for conversational FAQ) and validates the choice on a labeled set rather than guessing.

Likely follow-ups
  • How do you combine the two result lists?
Red-flag answer pattern
Insisting embeddings alone are always enough.
QWhat is an ANN index (HNSW) and what do you trade for speed?Intermediatetechnical deep-diveSeniorfreq: highBoth
Model answer

Searching millions of vectors exactly is too slow. An Approximate Nearest Neighbor (ANN) index like HNSW finds 'close enough' matches fast.

The trade-off: you give up a little accuracy (recall) for a big speed gain. A setting like ef_search lets you tune that balance — higher = more accurate but slower.

Strong senior-level answer

A senior picks the setting from a latency budget: raise accuracy until it stops improving within the latency you can afford, and also accounts for memory and re-index cost as the data grows.

Likely follow-ups
  • How does HNSW differ from IVF?
Red-flag answer pattern
Believing ANN search is exact/lossless.

QHow do you choose a vector database?Intermediatesystem designSeniorfreq: mediumBoth
Model answer

Pick based on scale, filtering needs, and how much infra you want to run:

  • pgvector — you already run Postgres and scale is moderate.
  • Qdrant / Weaviate / Milvus — large scale, heavy metadata filtering.
  • FAISS — in-process, offline, or research.
  • Managed (Pinecone etc.) — you don't want to operate infra.
Strong senior-level answer

A senior starts from constraints (data residency, multi-tenancy, can the team run another stateful service?) and avoids adopting a heavy vector DB before the scale actually requires it — pgvector covers a lot.

Likely follow-ups
  • Why not always use a managed service?
Red-flag answer pattern
Picking the trendiest DB with no reference to scale or operations.
QWhat does reranking add, and why a cross-encoder?Advancedtechnical deep-diveSeniorfreq: mediumBoth
Model answer

First-pass search returns many roughly-relevant chunks. A reranker reorders the top candidates by true relevance and keeps only the best few.

A cross-encoder reads the query and a chunk together (not separately), so it judges relevance much more precisely — at the cost of speed, which is why you only run it on the top ~50, not the whole corpus.

Strong senior-level answer

A senior uses reranking to keep the context small and precise (cheaper, less 'lost in the middle') and measures the recall@k improvement to justify it.

Likely follow-ups
  • Why not use a cross-encoder for the whole search?
Red-flag answer pattern
Stuffing many un-reranked chunks straight into the prompt.

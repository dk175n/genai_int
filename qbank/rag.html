QWalk me through a production RAG pipeline.Intermediatetechnical deep-diveSeniorfreq: highBoth
Model answer

RAG (Retrieval-Augmented Generation) grounds an LLM in your own data. It has two halves:

Offline (done ahead of time):
  • Load the documents and extract text.
  • Chunk them into smaller passages.
  • Embed each chunk into a vector.
  • Index the vectors in a vector database.
Online (each user question):
  • Retrieve the most relevant chunks (hybrid: semantic + keyword).
  • Rerank them and keep the best few.
  • Ground — put those chunks in the prompt with 'answer only from this context'.
  • Generate + cite — the model answers and cites which chunk each claim came from.
  • Validate the answer is supported before showing it.

Then evaluate both halves separately — retrieval quality and answer quality.

Strong senior-level answer

A senior adds the production concerns: access control so users only see permitted docs, caching for cost/latency, observability (logging the retrieved chunks per query), an 'I don't know' path, and running evaluation automatically on every change.

Likely follow-ups
  • Where does it most commonly fail?
  • How do you handle permissions / multi-tenant data?
  • How do you keep the index fresh?
Red-flag answer pattern
Describing only 'embed → cosine search → stuff into the prompt' with no reranking, citations or evaluation.

QRAG vs fine-tuning — when do you pick which?Intermediatetechnical deep-diveSeniorfreq: highBoth
Model answer
  • RAG — for knowledge that changes often, is large, or must be sourced/audited. Updating means re-indexing, not retraining.
  • Fine-tuning — for changing the model's behavior, format, tone or domain style.
  • They combine: fine-tune the how it talks, use RAG for the facts.
Strong senior-level answer

A senior defaults to RAG because updates are cheap and answers are auditable, and only fine-tunes when prompting + RAG can't hit a behavioral or format target — and even then prefers lightweight (LoRA) fine-tuning.

Likely follow-ups
  • Can you do both at once?
Red-flag answer pattern
'Just fine-tune the model on our documents' as the default for a knowledge problem.
QHow do you choose a chunking strategy?Advancedtechnical deep-diveSeniorfreq: highBoth
Model answer

Chunking = how you split documents. It's the highest-leverage RAG decision:

  • Too big → embeddings get diluted and you waste context space.
  • Too small → the answer gets split across chunks.
  • Start with fixed-size + ~10–20% overlap; move to splitting on document structure (headings/sections) for structured docs.
  • A common best option: 'parent-document' — match on small precise chunks but return the larger surrounding passage for context.
Strong senior-level answer

A senior treats chunking as an experiment driven by a labeled query set, checking whether answers span chunk boundaries, rather than picking a fixed number by habit.

Likely follow-ups
  • What overlap do you use and why?
Red-flag answer pattern
'Always 512 tokens' with no reference to the documents or any measurement.
QHow do you evaluate a RAG system?Advancedtechnical deep-diveSeniorfreq: highBoth
Model answer

Evaluate the two halves separately so you know which one failed:

  • Retrieval — did it find the right documents? Metrics: recall@k, MRR, on a labeled set of question→relevant-doc.
  • Generation — is the answer supported by the retrieved context (faithfulness), relevant, and correct vs a known answer?

Use a small 'golden' set of test cases, and an LLM-as-judge to score at scale.

Strong senior-level answer

A senior runs evaluation like CI (every change must not regress), grows the test set from real production failures, and calibrates the LLM judge against some human labels so they trust it.

Likely follow-ups
  • How big should the golden set be?
  • How do you trust an LLM judge?
Red-flag answer pattern
'We eyeballed a few answers and they looked good.'

QHow do you reduce hallucination in a RAG system?Advancedtechnical deep-diveSeniorfreq: highBoth
Model answer
  • Ground + cite or abstain — instruct the model to answer only from the provided context, cite sources, and say 'I don't have enough information' otherwise.
  • Better retrieval — hybrid search + reranking so the right context is actually present.
  • Coverage gate — if the best retrieved chunk scores too low, abstain or escalate instead of guessing.
  • Faithfulness check — verify the answer's claims are supported before showing it.
Strong senior-level answer

A senior makes hallucination a tracked metric with a deploy gate, rather than a vague hope, and samples production traffic to keep measuring it.

Likely follow-ups
  • What if retrieval returns nothing relevant?
Red-flag answer pattern
Trying to fix it purely with prompt wording like 'be accurate'.

System Design

Design · SQL Data Agent

A natural-language analytics agent over a data warehouse: schema-aware text-to-SQL with a semantic layer, read-only governance, and result validation. Scales the SQL agent project to an enterprise design.

Agent + governance

The brief

Text-to-SQL at organizational scale. The hard parts are correctness on a big/ambiguous schema (semantic layer helps), governance (read-only, row-level security), and trust in results.

Clarifying questions to ask first

  • Warehouse (Snowflake/BigQuery/Postgres)?
  • Is there a semantic layer / metric definitions?
  • Row-level security needs?
  • Who consumes results — analysts or end users?

Functional requirements

  • NL → governed SQL with results + the SQL shown
  • Use a semantic layer / approved metrics
  • Charts/summaries
  • Save/share queries

Non-functional requirements

  • Read-only + RLS enforced
  • Query cost/row limits
  • Correctness on the schema
  • Auditability

Architecture

Question
Schema + semantic layer
Generate governed SQL
Validate + RLS + limits
Execute + summarize

Data ingestion layer

Index schema, column descriptions, and metric/semantic definitions; embed example question→SQL pairs for few-shot.

Retrieval layer

Retrieve relevant tables/columns/metrics for the question to ground SQL generation (RAG over the schema).

Model / orchestration layer

Generate SQL grounded in retrieved schema + semantic layer; self-check against schema; optionally repair on execution error (bounded).

Safety layer

Read-only role + row-level security; validate SELECT-only; enforce LIMIT/cost guards; parameterize; show SQL for transparency; audit every query.

Evaluation layer

Execution accuracy vs golden question→result set; safety (no unsafe/over-cost query passes); track repair rate.

Observability layer

Trace question→SQL→result; monitor accuracy, query cost, error/repair rate; alert on cost spikes.

Cost & latency tradeoffs

LeverEffectTradeoff
Semantic layer / approved metrics↑ correctnessUpfront modeling effort
Cost/row limits↓ runaway costTruncated results

Failure modes

  • Wrong joins/metrics → semantic layer + schema RAG + validation
  • Runaway/expensive query → cost guards + LIMIT
  • RLS bypass → enforce at DB role, not prompt
  • Hallucinated columns → ground + validate

How to explain this design in 10 minutes

Clarify warehouse + semantic layer + governance; draw schema-RAG + governed generation + validation + RLS; stress read-only governance and correctness; cover accuracy eval; close with cost/failure modes.

Common interviewer interruptions & how to answer

Interviewer says…How to answer
“What if retrieval returns nothing relevant?”Coverage gate → abstain or escalate; never force an answer. Measure abstain-correctness.
“This will be too expensive.”Cache + route to a smaller model for easy traffic; show cost/req and that quality held on eval.
“How do you know it's any good?”Golden set offline as a deploy gate + sampled online judging; split retrieval vs generation metrics.
“What about data the user shouldn't see?”Access control enforced at retrieval, not the prompt; per-tenant filters or separate indexes.
“What happens when the model provider goes down?”Timeouts, fallback model/provider, graceful degradation to cached/smaller answers.

Interview questions on this design (3)

QHow do you get correct SQL on a large, ambiguous schema?Advancedsystem designSeniorfreq: mediumBoth
Model answer

RAG over the schema + column/metric descriptions to ground generation, a semantic layer of approved metrics, few-shot example pairs, and a self-check/repair loop against execution errors. Validate before running.

Strong senior-level answer

The semantic layer is the highest-leverage investment — it constrains the model to correct, governed definitions and removes most ambiguity. I measure execution accuracy on a golden set and treat governance (read-only, RLS) as non-negotiable.

Likely follow-ups
  • How do you enforce row-level security?
  • How do you handle ambiguous metric definitions?
Red-flag answer pattern
Free text-to-SQL with no schema grounding or governance.
QHow do you enforce governance (read-only, row-level security) on a data agent?Seniorsystem designSenior/Stafffreq: mediumBoth
Model answer

Enforce at the database, not the prompt: a read-only role, row-level security tied to the requesting user, query validation (SELECT-only), and cost/row limits. Show the generated SQL for transparency and audit-log every query.

Strong senior-level answer

A senior makes unsafe queries impossible by construction (least-privilege role) so governance holds regardless of what the model emits, and attributes query cost back to users.

Likely follow-ups
  • What if a user needs write access?
Red-flag answer pattern
Relying on the prompt to keep the agent read-only.
QHow do you stop runaway-cost queries on a warehouse?Advancedsystem designSeniorfreq: mediumBoth
Model answer

Enforce LIMITs, query timeouts, and scanned-bytes/cost caps; validate before executing; prefer a semantic layer with pre-approved, bounded queries. Monitor and alert on cost spikes.

Strong senior-level answer

A senior sets a per-query cost ceiling and a dry-run/estimate step, because a single bad generated query can scan terabytes on systems billed by data scanned.

Likely follow-ups
  • How do you estimate cost before running?
Red-flag answer pattern
No limits, letting a generated query scan the whole warehouse.

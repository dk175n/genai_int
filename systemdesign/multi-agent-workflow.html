Design · Multi-Agent Business Workflow

Multiple specialized agents coordinating a business process (e.g. intake → research → draft → review) with a supervisor, shared state, and human checkpoints. When and how to actually use multi-agent.

Advanced

The brief

A coordinated set of agents. The senior move is justifying multi-agent at all (vs one agent or a workflow) and designing the coordination, shared state and human checkpoints so it's reliable, not chaotic.

Clarifying questions to ask first

  • What are the distinct steps/roles?
  • Are steps fixed (→ workflow) or dynamic (→ agents)?
  • Where must a human approve?
  • What's the SLA and cost ceiling?

Functional requirements

  • Decompose a process into specialized agents
  • Supervisor/orchestrator routes and aggregates
  • Shared, checkpointed state
  • Human approval at key steps
  • Audit of decisions

Non-functional requirements

  • Bounded total cost/steps
  • Observability across agents
  • Reliability (retry a step, not the whole run)
  • Clear failure isolation

Architecture

Request
Supervisor / planner
Specialist agentsresearch·draft·check
Shared state + checkpoints
Human approval → output

Data ingestion layer

Per-agent knowledge/tools as needed (some agents are RAG-backed, some tool-backed).

Retrieval layer

Each specialist retrieves within its domain; shared state carries intermediate results, not raw context dumps.

Model / orchestration layer

Stateful graph (LangGraph-style): supervisor node routes to specialists; explicit shared state; checkpointing for retry/resume; human-in-the-loop nodes for approval. Bound total steps/cost.

Safety layer

Per-agent least privilege; guardrails at each boundary; human approval before external/irreversible actions; treat inter-agent messages as untrusted.

Evaluation layer

End-to-end process success rate + per-agent quality; cost/steps per run; track where runs fail (which agent/step).

Observability layer

Trace the whole graph (which agent did what, state transitions); monitor success rate, step count, cost, human-intervention rate.

Cost & latency tradeoffs

LeverEffectTradeoff
Fewer agents / collapse to workflow↓ cost & failure surfaceLess flexibility
Parallel independent agents↓ latencyCoordination complexity

Failure modes

  • Coordination chaos / loops → supervisor + bounded steps + checkpoints
  • Error cascade → isolate + retry single step
  • Cost blowup → global budget
  • Over-engineering → justify vs single agent/workflow

How to explain this design in 10 minutes

First justify multi-agent vs a workflow; draw supervisor + specialists + shared state + human checkpoints; stress bounded cost, checkpointing and observability; cover end-to-end eval; close by noting where you'd simplify.

Common interviewer interruptions & how to answer

Interviewer says…How to answer
“What if retrieval returns nothing relevant?”Coverage gate → abstain or escalate; never force an answer. Measure abstain-correctness.
“This will be too expensive.”Cache + route to a smaller model for easy traffic; show cost/req and that quality held on eval.
“How do you know it's any good?”Golden set offline as a deploy gate + sampled online judging; split retrieval vs generation metrics.
“What about data the user shouldn't see?”Access control enforced at retrieval, not the prompt; per-tenant filters or separate indexes.
“What happens when the model provider goes down?”Timeouts, fallback model/provider, graceful degradation to cached/smaller answers.

Interview questions on this design (3)

QWhen is multi-agent actually worth it over a single agent or a workflow?Seniorsystem designSenior/Stafffreq: emergingBoth
Model answer

When the process has genuinely distinct skills/roles and dynamic routing between them; otherwise a single agent or deterministic workflow is cheaper and more reliable. Multi-agent adds coordination cost and failure surface.

Strong senior-level answer

I default to the simplest thing: fixed steps → workflow; one dynamic skill → one agent; multiple distinct dynamic skills with routing → multi-agent with a supervisor, shared state and checkpoints. I'd explicitly say where I'd collapse agents to reduce failure surface — that restraint is the signal.

Likely follow-ups
  • How do you stop agents looping between each other?
  • How do you checkpoint shared state?
Red-flag answer pattern
Reaching for multi-agent because it's impressive, with no justification.
QHow does the supervisor coordinate agents and manage shared state?Advancedsystem designSenior/Stafffreq: emergingBoth
Model answer

A supervisor (orchestrator) routes work to specialist agents, aggregates results, and decides next steps. Shared state is explicit and checkpointed (a graph), passing intermediate results — not raw context dumps — between agents.

Strong senior-level answer

A senior bounds total steps/cost globally, checkpoints so a single agent can be retried without re-running everything, and traces across agents so failures are attributable.

Likely follow-ups
  • How do you bound total cost across all agents?
Red-flag answer pattern
Agents passing full conversation history to each other with no shared-state design.
QWhere do you put human-in-the-loop in a multi-agent business workflow?Seniorsystem designSenior/Stafffreq: mediumBoth
Model answer

At irreversible or high-impact steps: before external sends, financial actions, or publishing. The graph pauses at a human-approval node, then resumes from the checkpoint.

Strong senior-level answer

A senior maps approval points to business risk (not every step), and designs the resume-from-checkpoint flow so waiting for a human doesn't lose state or block unrelated work.

Likely follow-ups
  • How do you resume cleanly after approval?
Red-flag answer pattern
Full autonomy on irreversible business actions with no approval gate.

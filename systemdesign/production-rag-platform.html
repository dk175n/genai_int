System Design

Design · Production RAG Platform

Not one app — the shared platform many RAG apps run on: ingestion service, retrieval service, model gateway, eval harness, and observability, multi-tenant. The platform-engineer design.

Platform/Staff

The brief

A reusable internal platform so product teams ship RAG/agent features without rebuilding ingestion, retrieval, gateways, eval and observability each time. Multi-tenancy, standardization and self-serve are the themes.

Clarifying questions to ask first

  • How many teams/use-cases will onboard?
  • Shared vs isolated data per tenant?
  • Self-serve or platform-team-managed onboarding?
  • What standards must every app meet (eval, guardrails)?

Functional requirements

  • Ingestion-as-a-service (connectors, chunk, embed, index)
  • Retrieval API (hybrid + rerank)
  • Model gateway (routing, caching, fallback, rate limits)
  • Shared eval harness + golden-set tooling
  • Central observability

Non-functional requirements

  • Multi-tenant isolation
  • Self-serve onboarding
  • Cost attribution per tenant
  • Reliability/SLOs
  • Enforced guardrail/eval standards

Data ingestion layer

Ingestion service with pluggable connectors + standardized chunk/embed/index; per-tenant namespaces; incremental sync; tracks lineage.

Retrieval layer

Retrieval API: hybrid + rerank with tenant-scoped indexes/filters; tunable per app; consistent metrics.

Model / orchestration layer

Model gateway: routing (small/large), caching (exact + semantic), fallback providers, rate limiting, prompt registry/versioning, cost attribution.

Safety layer

Centralized guardrail middleware (injection, PII, policy) every app inherits; tenant isolation enforced at retrieval and gateway.

Evaluation layer

Shared eval harness: golden-set management, LLM-judge library, CI integration; every onboarded app must pass baseline eval + guardrail standards.

Observability layer

Central tracing + dashboards per tenant: cost, latency, quality, drift; cost attribution and quota alerts.

Failure modes

  • Noisy neighbor → per-tenant quotas/rate limits
  • Cross-tenant leak → strict namespace isolation + cache keys
  • Gateway is SPOF → HA + fallback
  • Teams bypass standards → enforce via the platform, not docs

How to explain this design in 10 minutes

Frame as platform not app; draw gateway + retrieval svc + ingestion svc + eval + observability, multi-tenant; stress isolation, self-serve, cost attribution and enforced standards; cover platform-level SLOs; close with multi-tenant failure modes.

Common interviewer interruptions & how to answer

Interviewer says…How to answer
“What if retrieval returns nothing relevant?”Coverage gate → abstain or escalate; never force an answer. Measure abstain-correctness.
“This will be too expensive.”Cache + route to a smaller model for easy traffic; show cost/req and that quality held on eval.
“How do you know it's any good?”Golden set offline as a deploy gate + sampled online judging; split retrieval vs generation metrics.
“What about data the user shouldn't see?”Access control enforced at retrieval, not the prompt; per-tenant filters or separate indexes.
“What happens when the model provider goes down?”Timeouts, fallback model/provider, graceful degradation to cached/smaller answers.

Interview questions on this design (3)

QHow do you design a multi-tenant RAG platform that enforces quality and isolation?Seniorsystem designStafffreq: emergingBoth
Model answer

Shared ingestion/retrieval/gateway services with per-tenant namespaces and quotas; a model gateway for routing/caching/fallback/cost-attribution; a shared eval harness and guardrail middleware that every onboarded app must pass; central observability. Isolation enforced at retrieval and cache keys.

Strong senior-level answer

I'd make standards executable: an app can't onboard without passing baseline eval and inheriting guardrails — enforcement lives in the platform, not a wiki. I design for cost attribution and noisy-neighbor control from day one, and keep the gateway HA since it's the critical path.

Likely follow-ups
  • How do you attribute cost per tenant?
  • How do you prevent a shared cache leaking across tenants?
Red-flag answer pattern
A single shared index for all tenants with no isolation story.
QWhat does the model gateway do, and why centralize it?Seniorsystem designStafffreq: emergingBoth
Model answer

A single entry point for all model calls that handles routing (small vs large), caching, provider fallback, rate limiting, prompt-version registry, and cost attribution per tenant/feature. Centralizing means every app inherits these without rebuilding them.

Strong senior-level answer

A senior makes the gateway highly available (it's the critical path), and uses it to enforce org-wide policy — no app talks to a provider directly, so guardrails, caching and cost controls are guaranteed.

Likely follow-ups
  • Isn't the gateway a single point of failure?
Red-flag answer pattern
Each team calling providers directly with duplicated, inconsistent logic.
QHow do you enforce quality and guardrail standards across many tenant apps?Seniorsystem designStafffreq: emergingBoth
Model answer

Make standards executable, not documentation: a shared eval harness and guardrail middleware that every app must pass/inherit to onboard. An app can't go live without meeting a baseline faithfulness and passing adversarial/injection checks.

Strong senior-level answer

A senior puts enforcement in the platform (CI gate + inherited middleware) rather than a wiki, so standards can't be skipped, and tracks per-tenant quality centrally.

Likely follow-ups
  • What's your onboarding gate?
Red-flag answer pattern
Relying on teams to 'follow the guidelines' voluntarily.

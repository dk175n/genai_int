System Design

Design · Enterprise Knowledge Assistant

“Build an assistant that answers employee questions over our internal knowledge (wiki, docs, tickets), with citations and respecting permissions.” The canonical enterprise RAG design — run it with the framework.

Most common design QSenior/Staff

The brief

A read-mostly RAG system over heterogeneous internal sources. The defining constraints are access control (employees must only see what they're entitled to), freshness (sources change daily), and trust (citations + abstain).

Clarifying questions to ask first

  • Which sources, and how do their permissions work (per-doc ACLs? groups)?
  • How fresh must answers be — minutes or daily?
  • Expected scale: #docs, #users, QPS?
  • Quality bar and consequence of a wrong answer?
  • On-prem / data-residency / compliance constraints?
  • Do we need conversational follow-ups or single-shot Q&A?

Functional requirements

  • Answer NL questions over internal sources with citations
  • Respect per-user permissions
  • Abstain/escalate when unsure
  • Conversational follow-ups
  • Capture feedback (thumbs + reason)
  • Admin view of unanswered/escalated questions (KB gaps)

Non-functional requirements

  • p95 latency target (e.g. < 4s)
  • Cost/query budget
  • Groundedness/faithfulness target (e.g. ≥ 0.9 on golden set)
  • No cross-permission data leakage (hard requirement)
  • Index freshness SLA
  • Auditability of every answer

Architecture

Connectorswiki·docs·tickets
Ingestionchunk·embed·ACL metadata
Retrievalhybrid·rerank·ACL filter
Orchestrationground·cite·abstain
Safety + Eval + Obs

Data ingestion layer

Connectors per source (with incremental sync). Parse → structure-aware chunk → embed → store with rich metadata: source, URL, last-modified, and ACL fields (owner group / visibility) carried from the source system. Incremental re-index on change events or a schedule to meet the freshness SLA; tombstone deleted docs so they leave the index.

Retrieval layer

Hybrid (dense + BM25) + cross-encoder rerank. Access control is applied as a metadata filter on the query, tied to the requesting user's groups — restricted docs are never candidates. For hard boundaries, separate indexes per security domain. Return top-k with source metadata for citations.

Model / orchestration layer

Grounded prompt: context-only answers, cite sources, abstain if coverage is weak. Single capable model; optionally route trivial FAQ to a cheaper model. Conversation memory bounded (summarize old turns). No tools needed for v1 (read-only Q&A).

Safety layer

Input: scope/PII checks, injection detection. Retrieval: ACL enforcement (the key control). Output: groundedness check, PII/secret filter, citation presence. Treat retrieved content as untrusted (indirect injection from a malicious doc).

Evaluation layer

Golden set per source/domain (question → answer + allowed sources). Offline gate on faithfulness + recall@k + abstain-correctness; CI on every prompt/retrieval change. Online: sample traffic for faithfulness judging; track thumbs-down and escalations as new eval cases and KB gaps.

Observability layer

Trace each query (retrieved chunks + ACL filter applied + tokens + cost + latency). Dashboards: cost/query, p95 latency, faithfulness, abstain/escalation rate, freshness lag. Alerts on quality regressions and any ACL-filter anomaly.

Failure modes

  • ACL filter bug → cross-permission leak (highest severity; test relentlessly)
  • Stale index → wrong/outdated answers (freshness SLA + tombstoning)
  • Bad retrieval → hallucination (hybrid+rerank, abstain gate)
  • Indirect injection from a poisoned doc → untrusted content, output guard
  • Provider outage → fallback model + cached answers

How to explain this design in 10 minutes

Minute 1–2: clarify sources, permissions, freshness, scale, quality bar. Minute 3–5: draw the five layers; emphasize ACL-at-retrieval and citations. Minute 6–8: evaluation + observability as first-class. Minute 9–10: tradeoffs (cost/latency, freshness vs cache) and the top failure modes with mitigations. Keep returning to the permission requirement — it's what makes this 'enterprise'.

Common interviewer interruptions & how to answer

Interviewer says…How to answer
“What if retrieval returns nothing relevant?”Coverage gate → abstain or escalate; never force an answer. Measure abstain-correctness.
“This will be too expensive.”Cache + route to a smaller model for easy traffic; show cost/req and that quality held on eval.
“How do you know it's any good?”Golden set offline as a deploy gate + sampled online judging; split retrieval vs generation metrics.
“What about data the user shouldn't see?”Access control enforced at retrieval, not the prompt; per-tenant filters or separate indexes.
“What happens when the model provider goes down?”Timeouts, fallback model/provider, graceful degradation to cached/smaller answers.

Interview questions on this design (2)

QHow do you guarantee an employee never sees a document they're not allowed to?Seniorsystem designSenior/Stafffreq: highBoth
Model answer

Enforce access control at retrieval via metadata filters tied to the user's groups (and/or separate indexes per security domain), so restricted docs are never retrieved. Add an output secret/PII backstop. Never rely on the prompt.

Strong senior-level answer

I carry ACLs from the source system into chunk metadata at ingestion, filter on them in the vector query, and test it as a hard invariant (red-team queries that should return nothing). For strict isolation I use per-domain indexes. The model is the last line I'd trust for permissions.

Likely follow-ups
  • Per-doc ACLs vs separate indexes — when each?
  • How do ACL changes propagate to the index?
Red-flag answer pattern
Telling the model not to reveal restricted info.
QHow do you keep answers fresh as the wiki changes constantly?Advancedsystem designSeniorfreq: mediumBoth
Model answer

Incremental sync on change events (or scheduled), re-embed changed docs, tombstone deletes, and invalidate caches on re-index. Define a freshness SLA and monitor index lag.

Strong senior-level answer

I treat ingestion as a pipeline with its own monitoring (lag, failures) because stale answers erode trust silently. Cache TTLs are tied to the freshness SLA, and I surface 'last updated' in citations so users can judge recency.

Likely follow-ups
  • How do you handle a deleted/again-restricted doc still in cache?
Red-flag answer pattern
Full nightly re-index of everything with no incremental path at scale.

System Design

Design · Code Assistant Agent

An agent that answers questions and makes changes across a codebase: repo-aware retrieval, tools (read/search/edit/run tests) in a sandbox, with guardrails on execution.

Agent + RAG

The brief

RAG over code + an action-taking agent. Retrieval must be code-aware (symbols, files), and execution (running tests, editing) must be sandboxed and bounded.

Clarifying questions to ask first

  • Read-only Q&A or does it edit code?
  • Can it run code/tests? In what sandbox?
  • Repo size and languages?
  • How are changes reviewed (PR + human)?

Functional requirements

  • Answer questions about the repo with file/line citations
  • Propose edits as diffs
  • Run tests in a sandbox
  • Open a PR for human review

Non-functional requirements

  • No unsandboxed execution
  • Bounded cost/steps per task
  • Latency tolerable for dev workflow
  • No secret leakage from repo

Architecture

Task
Code-aware retrievesymbols·files
Plan + edit (diff)
Run tests (sandbox)
PR for review

Data ingestion layer

Index repo by file + symbol; chunk along function/class boundaries; keep path metadata; re-index on commit.

Retrieval layer

Hybrid + symbol search; expand to surrounding file context (parent-doc style).

Model / orchestration layer

Planner-executor or graph: locate → plan edit → apply diff → run tests → iterate (bounded). Tools: search, read, edit, run_tests.

Safety layer

Execution only in an ephemeral sandbox; no network/secrets; bounded steps & cost; all changes via PR (human merges); treat repo content as untrusted (injection in comments).

Evaluation layer

Task success = tests pass / correct diff on a benchmark of tasks; track steps and cost; regression on a fixed task set.

Observability layer

Trace plan + tool calls + diffs; monitor task success rate, sandbox failures, cost/task.

Cost & latency tradeoffs

LeverEffectTradeoff
Scope retrieval to relevant files↓ cost/latencyRisk missing context
Cap iterations↓ costMay not finish hard tasks

Failure modes

  • Unsafe execution → sandbox + no secrets
  • Endless edit-test loops → step budget
  • Bad edits merged → PR + human review
  • Injection via code comments → untrusted content

How to explain this design in 10 minutes

Clarify edit-vs-readonly and execution; draw code-aware retrieval + planner-executor + sandbox + PR gate; stress sandboxing and bounded loops; cover task-success eval; close with failure modes.

Common interviewer interruptions & how to answer

Interviewer says…How to answer
“What if retrieval returns nothing relevant?”Coverage gate → abstain or escalate; never force an answer. Measure abstain-correctness.
“This will be too expensive.”Cache + route to a smaller model for easy traffic; show cost/req and that quality held on eval.
“How do you know it's any good?”Golden set offline as a deploy gate + sampled online judging; split retrieval vs generation metrics.
“What about data the user shouldn't see?”Access control enforced at retrieval, not the prompt; per-tenant filters or separate indexes.
“What happens when the model provider goes down?”Timeouts, fallback model/provider, graceful degradation to cached/smaller answers.

Interview questions on this design (3)

QHow do you safely let an agent run and modify code?Seniorsystem designSeniorfreq: emergingBoth
Model answer

Run only in an ephemeral, network/secret-free sandbox; bound steps and cost; surface changes as diffs and require a human-reviewed PR to merge; treat repo content as untrusted.

Strong senior-level answer

Safety comes from the environment, not the model: the sandbox can't reach secrets or prod, the agent can't merge, and every action is traced. I'd benchmark task success on a fixed set so improvements are measured.

Likely follow-ups
  • How do you bound cost on hard tasks?
  • How is repo content an injection vector?
Red-flag answer pattern
Letting the agent execute with prod credentials or auto-merge.
QHow do you do code-aware retrieval over a large repo?Advancedsystem designSeniorfreq: emergingBoth
Model answer

Index by file and symbol (functions/classes), chunk along code boundaries rather than fixed sizes, keep path metadata, and re-index on commit. Combine semantic search with symbol/keyword lookup, then expand to surrounding file context.

Strong senior-level answer

A senior scopes retrieval to the relevant files/modules to keep context small and cheap, and notes that code has strong structure (imports, call graphs) you can exploit beyond plain text chunking.

Likely follow-ups
  • How do you keep the index fresh on every commit?
Red-flag answer pattern
Treating code as plain prose and fixed-size chunking it.
QHow would you evaluate a code assistant agent?Advancedsystem designSeniorfreq: mediumBoth
Model answer

Benchmark of tasks with objective checks: do the generated tests pass / does the diff produce correct behavior? Track task success rate, steps and cost per task, and sandbox failure rate.

Strong senior-level answer

A senior uses tests-pass as the ground-truth signal (objective, unlike prose), and runs the benchmark as CI so model/prompt changes don't regress task success.

Likely follow-ups
  • What's your ground-truth signal?
Red-flag answer pattern
Judging by whether the code 'looks reasonable'.

System Design

Design · Customer Support Bot

A customer-facing support assistant over a help center: answer with citations, escalate when unsure, integrate with ticketing. Trust and deflection are the design axes.

Design round

The brief

Customer-facing RAG. Differs from the internal assistant in that users are external (higher injection/abuse risk), wrong answers hurt the brand, and it must hand off to human agents cleanly.

Clarifying questions to ask first

  • Channels (web chat, email, in-app)?
  • Tone/brand constraints?
  • Ticketing system to integrate with?
  • What's the cost of a wrong answer vs an escalation?
  • Multilingual?

Functional requirements

  • Answer from KB with citations
  • Escalate to human + create ticket with context
  • Multichannel
  • Capture CSAT/feedback
  • Surface KB gaps

Non-functional requirements

  • Low latency (live chat)
  • High groundedness (brand risk)
  • Abuse/injection resistance
  • Cost/conversation budget

Architecture

Data ingestion layer

Help-center articles + past resolved tickets → chunk → embed → index with product/category metadata and 'last reviewed' date.

Retrieval layer

Hybrid + rerank, scoped to the product/locale. Coverage gate decides answer vs escalate.

Model / orchestration layer

Grounded, on-brand prompt with citations and abstain. Bias toward escalation when confidence is low. Bounded conversation memory.

Safety layer

Stronger input guards (external users): abuse, jailbreak, injection. Output: groundedness, no internal-only info, policy/brand checks.

Evaluation layer

Golden set + deflection metric (correct auto-answer vs correct escalation) + faithfulness; A/B on escalation threshold.

Observability layer

Trace + CSAT, deflection rate, escalation precision, cost/conversation; alert on faithfulness drops.

Failure modes

  • Confident wrong answer → coverage gate, bias to escalate
  • External injection/abuse → input guards
  • Bad handoff loses context → pass transcript+retrieval to ticket
  • KB stale → review dates + re-index

How to explain this design in 10 minutes

Clarify channels + cost-of-wrong-answer; draw RAG + coverage gate + escalation; stress external-user safety; cover deflection/CSAT eval and the human handoff; close with tradeoffs.

Common interviewer interruptions & how to answer

Interviewer says…How to answer
“What if retrieval returns nothing relevant?”Coverage gate → abstain or escalate; never force an answer. Measure abstain-correctness.
“This will be too expensive.”Cache + route to a smaller model for easy traffic; show cost/req and that quality held on eval.
“How do you know it's any good?”Golden set offline as a deploy gate + sampled online judging; split retrieval vs generation metrics.
“What about data the user shouldn't see?”Access control enforced at retrieval, not the prompt; per-tenant filters or separate indexes.
“What happens when the model provider goes down?”Timeouts, fallback model/provider, graceful degradation to cached/smaller answers.

Interview questions on this design (3)

QHow do you balance deflection (auto-answering) against giving wrong answers?Seniorsystem designSeniorfreq: highBoth
Model answer

Set a coverage/faithfulness gate and bias toward escalation; measure escalation precision and CSAT alongside deflection so you don't optimize deflection at the cost of trust.

Strong senior-level answer

I treat a confident wrong answer as more expensive than an escalation and tune thresholds accordingly, A/B testing the threshold on real traffic. Every escalation feeds the KB-gap and eval loop.

Likely follow-ups
  • How do you hand off to a human without losing context?
Red-flag answer pattern
Maximizing deflection regardless of correctness.
QHow do you defend a customer-facing bot against abuse and prompt injection?Seniorsystem designSeniorfreq: highBoth
Model answer

External users mean stronger input guards: rate limiting, abuse/jailbreak detection, and injection checks. Treat the KB content as untrusted too (indirect injection). Output guard blocks internal-only info, PII and off-brand content before replying.

Strong senior-level answer

A senior assumes hostile input by default, keeps the bot scoped to a narrow domain (out-of-scope → escalate), and limits blast radius since it's internet-facing.

Likely follow-ups
  • How do you stop it answering off-topic questions?
Red-flag answer pattern
Treating an external bot with the same trust as an internal tool.
QWhat's your evaluation and rollout plan for this bot?Advancedsystem designSeniorfreq: mediumBoth
Model answer

Golden set + deflection and escalation-precision metrics offline; shadow/canary on real traffic; A/B the escalation threshold; monitor faithfulness and CSAT. Roll out gradually, gated on quality.

Strong senior-level answer

A senior ships behind a canary with a kill-switch back to human-only, and treats every escalation/thumbs-down as a new eval case and KB-gap.

Likely follow-ups
  • How do you A/B the escalation threshold safely?
Red-flag answer pattern
Big-bang launch with no canary or rollback.
